Why Is This an Issue?

CWE-601 flags code that redirects users to a URL derived from untrusted input without validation. An attacker crafts a link that looks legitimate (it starts with your domain) but redirects the victim to a malicious site — perfect for phishing campaigns, credential theft, and OAuth token interception.

Open redirects are dangerous because they:

  • Enable phishing — victims trust the initial domain and don't notice the redirect to an attacker-controlled lookalike site
  • Steal OAuth tokens — manipulating redirect_uri parameters to exfiltrate authorization codes
  • Bypass security filters — WAFs and URL scanners see the legitimate domain and allow the link through
  • Chain with other attacks — open redirects are commonly paired with SSRF, XSS, or CSRF to amplify impact

This weakness is part of OWASP A01:2021 — Broken Access Control. The primary defense is allowlisting known redirect targets and never forwarding users to arbitrary, user-supplied URLs. If dynamic redirects are required, validate that the target is a relative path or belongs to an explicit list of trusted domains.

Examples of the Issue

Dangerous: Unvalidated redirect from request parameters (Java / Servlet / Spring)

Passing a user-supplied url or returnTo parameter directly to response.sendRedirect() or Spring's redirect: prefix allows an attacker to craft a link like https://yourapp.com/login?returnTo=https://evil.com/phish that silently sends the victim to a phishing page after login.

// DANGEROUS: User-controlled redirect target — open redirect. // --- Servlet: redirect parameter used directly --- @WebServlet("/login") public class LoginServlet extends HttpServlet { protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws IOException { // ... authenticate user ... String returnTo = req.getParameter("returnTo"); // Attacker sets returnTo=https://evil.com/steal-creds resp.sendRedirect(returnTo); // ↑ Open redirect — no validation! } } // --- Spring MVC: redirect with user-controlled URL --- @Controller public class AuthController { @GetMapping("/oauth/callback") public String oauthCallback(@RequestParam String redirectUrl) { // Attacker manipulates redirectUrl to exfiltrate OAuth tokens. return "redirect:" + redirectUrl; // ↑ Open redirect — arbitrary external URL accepted. } } // --- Spring Boot: another common pattern --- @GetMapping("/goto") public void goTo(@RequestParam String url, HttpServletResponse resp) throws IOException { resp.sendRedirect(url); // No allowlist, no validation. }

Dangerous: Unvalidated redirect in Express / Node.js

Using res.redirect() with a query parameter or request body value allows attackers to redirect users to any external URL. Common patterns include post-login redirects, link shorteners, and OAuth callback handlers. Attackers also use protocol-relative URLs (//evil.com) and URL-encoded payloads to bypass naive checks.

// DANGEROUS: User-controlled redirect — open redirect in Express. const express = require('express'); const app = express(); // Post-login redirect — attacker controls "next" param. app.post('/login', (req, res) => { // ... authenticate user ... const next = req.query.next || '/dashboard'; // Attacker link: /login?next=https://evil.com/phish res.redirect(next); // ↑ Open redirect — no validation! }); // Link shortener / tracking redirect — no allowlist. app.get('/go', (req, res) => { const target = req.query.url; res.redirect(target); // Arbitrary URL accepted. }); // OAuth callback — redirect_uri not validated. app.get('/auth/callback', (req, res) => { const { code, redirect_uri } = req.query; // ... exchange code for token ... res.redirect(redirect_uri); // ↑ Attacker exfiltrates OAuth token via crafted redirect_uri. }); // Bypasses that fool naive prefix checks: // //evil.com — protocol-relative URL // /\evil.com — backslash interpreted as path by some servers // https://yourapp.com@evil.com — userinfo component trick // https://evil.com#yourapp.com — fragment confusion

Dangerous: Weak validation that is easily bypassed (Java & JavaScript)

Many developers add a startsWith() or contains() check thinking it prevents open redirects. Attackers routinely bypass these with subdomains, userinfo tricks, and URL encoding. A denylist or string-prefix approach is not sufficient.

// DANGEROUS: Weak validation — trivially bypassed. // --- Java: startsWith check bypassed --- String redirectUrl = req.getParameter("url"); if (redirectUrl.startsWith("https://myapp.com")) { resp.sendRedirect(redirectUrl); } // Bypass: https://myapp.com.evil.com/phish (subdomain trick) // Bypass: https://myapp.com@evil.com (userinfo trick) // --- JavaScript: includes() check bypassed --- const url = req.query.next; if (url.includes('myapp.com')) { res.redirect(url); } // Bypass: https://evil.com/?fake=myapp.com (query param) // Bypass: https://myapp.com.attacker.com (subdomain trick) // --- JavaScript: regex without anchoring --- if (/myapp\.com/.test(url)) { res.redirect(url); } // Bypass: https://evil.com/myapp.com (path component) // Bypass: https://notmyapp.com (partial match)

How to Resolve

Fix (Java): Allowlist trusted domains and enforce relative paths

The safest approach is to only allow relative paths or redirect to a predefined allowlist of trusted URLs. Parse the URL with java.net.URI to reliably extract the host, then compare against the allowlist. Never trust string-prefix checks alone — always parse the URL properly.

// SAFE: Allowlist-based redirect validation. import java.net.URI; import java.net.URISyntaxException; import java.util.Set; public class SafeRedirectUtil { // Allowlist of trusted redirect domains. private static final Set<String> ALLOWED_HOSTS = Set.of( "myapp.com", "www.myapp.com", "accounts.myapp.com" ); private static final String DEFAULT_REDIRECT = "/dashboard"; /** * Validates a redirect URL and returns a safe target. * Only allows relative paths or URLs to allowlisted hosts. */ public static String validateRedirect(String redirectUrl) { if (redirectUrl == null || redirectUrl.isBlank()) { return DEFAULT_REDIRECT; } // Allow relative paths (must start with / but NOT //). if (redirectUrl.startsWith("/") && !redirectUrl.startsWith("//")) { return redirectUrl; } // Parse and validate absolute URLs. try { URI uri = new URI(redirectUrl); String host = uri.getHost(); String scheme = uri.getScheme(); if (host != null && ALLOWED_HOSTS.contains(host.toLowerCase()) && ("https".equals(scheme) || "http".equals(scheme))) { return redirectUrl; } } catch (URISyntaxException e) { // Malformed URL — fall through to default. } return DEFAULT_REDIRECT; } } // --- Usage in Servlet --- String target = SafeRedirectUtil.validateRedirect( req.getParameter("returnTo") ); resp.sendRedirect(target); // --- Usage in Spring MVC --- @GetMapping("/oauth/callback") public String callback(@RequestParam String redirectUrl) { String safe = SafeRedirectUtil.validateRedirect(redirectUrl); return "redirect:" + safe; }

Fix (JS): URL parsing, allowlist validation, and middleware guard

Use the built-in URL constructor to parse redirect targets — this correctly handles protocol-relative URLs, userinfo tricks, and encoding bypasses. Implement a reusable middleware that validates every redirect against an allowlist. Default to a safe fallback for any unrecognized target.

// SAFE: Allowlist-based redirect validation for Express / Node.js. const ALLOWED_HOSTS = new Set([ 'myapp.com', 'www.myapp.com', 'accounts.myapp.com' ]); const DEFAULT_REDIRECT = '/dashboard'; /** * Validates a redirect URL. * Allows relative paths and URLs to allowlisted hosts only. */ function validateRedirectUrl(url) { if (!url) return DEFAULT_REDIRECT; // Allow relative paths (starts with / but NOT //). if (url.startsWith('/') && !url.startsWith('//')) { return url; } try { const parsed = new URL(url); if (['http:', 'https:'].includes(parsed.protocol) && ALLOWED_HOSTS.has(parsed.hostname.toLowerCase())) { return url; } } catch { // Malformed URL — fall through to default. } return DEFAULT_REDIRECT; } // --- Express middleware for safe redirects --- function safeRedirect(req, res, next) { res.safeRedirect = (url) => { res.redirect(validateRedirectUrl(url)); }; next(); } const express = require('express'); const app = express(); app.use(safeRedirect); // Usage — always use res.safeRedirect() instead of res.redirect(). app.post('/login', (req, res) => { // ... authenticate user ... res.safeRedirect(req.query.next); // Attacker's ?next=https://evil.com → redirected to /dashboard. }); app.get('/auth/callback', (req, res) => { // ... exchange code for token ... res.safeRedirect(req.query.redirect_uri); // Only allowlisted redirect_uri values are honoured. });

Defense-in-Depth: Mapped / indirect redirects and interstitial pages

The most robust pattern avoids passing URLs entirely. Instead, the client sends an opaque key that the server maps to a URL server-side. If a direct URL must be used, show an interstitial warning page so users can confirm before navigating to an external site.

// SAFE: Mapped redirects — URL never leaves the server. // --- Java: server-side redirect map --- private static final Map<String, String> REDIRECT_MAP = Map.of( "dashboard", "/app/dashboard", "profile", "/app/profile", "billing", "https://billing.myapp.com/account" ); @GetMapping("/go/{key}") public String mappedRedirect(@PathVariable String key) { String target = REDIRECT_MAP.getOrDefault(key, "/app/dashboard"); return "redirect:" + target; } // Client calls: /go/billing → server resolves the real URL. // Attacker cannot inject an arbitrary URL because only keys are accepted. // --- JavaScript (Express): same pattern --- const redirectMap = { dashboard: '/app/dashboard', profile: '/app/profile', billing: 'https://billing.myapp.com/account' }; app.get('/go/:key', (req, res) => { const target = redirectMap[req.params.key] || '/app/dashboard'; res.redirect(target); }); // --- Interstitial page for user-submitted external links --- app.get('/external', (req, res) => { const url = req.query.url; // Show a warning page instead of redirecting immediately. res.render('external-warning', { externalUrl: url, message: 'You are leaving MyApp. Continue to external site?' }); // User must click "Continue" — no automatic redirect. });
