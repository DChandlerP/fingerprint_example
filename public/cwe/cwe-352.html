Why Is This an Issue?

CWE-352 flags web applications that process state-changing requests without verifying that the request was intentionally initiated by the authenticated user. In a CSRF attack, a malicious website tricks the victim's browser into sending a forged request — complete with the victim's session cookie — to the vulnerable application. Because the browser automatically attaches cookies, the server cannot distinguish legitimate requests from forged ones.

The impact depends on the targeted endpoint: CSRF can change passwords, transfer funds, modify account settings, or elevate privileges — all without the user's knowledge. The attack requires no special access to the application itself; the attacker only needs to lure the victim to a page they control (or inject content into a page the victim visits).

This weakness is ranked in the SANS/CWE Top 25 and maps to OWASP A01:2021 — Broken Access Control. The primary defense is the Synchronizer Token Pattern: embed a cryptographically random, per-session token in every state-changing form, and validate it server-side before processing the request.

Examples of the Issue

Dangerous: Java servlet processes POST without CSRF token verification

This servlet changes the user's email address based solely on the session cookie. An attacker hosts a hidden form that auto-submits a POST to /update-email — the victim's browser sends the session cookie automatically, and the email is changed without the user's knowledge or consent.

// DANGEROUS: No CSRF token — any cross-origin form submission is accepted. @WebServlet("/update-email") public class UpdateEmailServlet extends HttpServlet { protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException { HttpSession session = req.getSession(false); if (session == null) { resp.sendRedirect("/login"); return; } String userId = (String) session.getAttribute("userId"); String newEmail = req.getParameter("email"); // Attacker-controlled // No CSRF verification — the server trusts the cookie alone. userService.updateEmail(userId, newEmail); resp.getWriter().write("Email updated successfully."); } } <!-- Attacker's page performs the attack with a hidden auto-submitting form: --> <form action="https://vulnerable-app.com/update-email" method="POST"> <input type="hidden" name="email" value="attacker@evil.com" /> </form> <script>document.forms[0].submit();</script>

Dangerous: Express.js route trusts cookies alone for state-changing action

This Express route processes a password change by reading the session cookie and the new password from the POST body. There is no CSRF token or origin check. An attacker can forge a cross-origin POST using a hidden form or fetch() with credentials: 'include'.

// DANGEROUS: No CSRF token, no origin header check. const express = require('express'); const session = require('express-session'); const app = express(); app.use(express.urlencoded({ extended: true })); app.use(session({ secret: 'keyboard-cat', resave: false, saveUninitialized: false })); app.post('/change-password', (req, res) => { if (!req.session.userId) return res.status(401).send('Not logged in.'); const newPassword = req.body.password; // Attacker-controlled // No CSRF protection — session cookie is the only check. userService.changePassword(req.session.userId, newPassword); res.send('Password changed.'); });

Dangerous: Explicitly disabling Spring Security's built-in CSRF protection

Spring Security enables CSRF protection by default. Calling csrf().disable() turns it off entirely — a common but dangerous shortcut that developers often take during development and forget to revert. This leaves every POST, PUT, and DELETE endpoint vulnerable to CSRF.

// DANGEROUS: Disabling Spring Security's built-in CSRF protection. @Configuration @EnableWebSecurity public class SecurityConfig extends WebSecurityConfigurerAdapter { @Override protected void configure(HttpSecurity http) throws Exception { http .csrf().disable() // THIS IS THE PROBLEM — CSRF is now off globally .authorizeRequests() .antMatchers("/login", "/register").permitAll() .anyRequest().authenticated() .and() .formLogin() .loginPage("/login"); } }

How to Resolve

Fix (Java): Keep Spring Security's CSRF enabled + include token in forms

The simplest fix in Spring is to not disable CSRF. Spring Security automatically generates a per-session token stored in the HttpSession. Thymeleaf and Spring's form tag library inject it into forms automatically. For AJAX requests, read the token from a meta tag and include it as a header.

// SAFE: CSRF protection is ON (the default). Do NOT call csrf().disable(). @Configuration @EnableWebSecurity public class SecurityConfig extends WebSecurityConfigurerAdapter { @Override protected void configure(HttpSecurity http) throws Exception { http // CSRF is enabled by default — no need to call .csrf() at all. // If using cookie-based CSRF tokens (for SPA/AJAX): .csrf() .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()) .and() .authorizeRequests() .antMatchers("/login", "/register").permitAll() .anyRequest().authenticated() .and() .formLogin() .loginPage("/login"); } } // In Thymeleaf templates, the token is injected automatically: // <form th:action="@{/update-email}" method="post"> // <!-- Hidden _csrf field is added automatically by Thymeleaf --> // <input type="email" name="email" /> // <button type="submit">Update</button> // </form> // For AJAX requests, read the token from the cookie: // const token = document.cookie.match(/XSRF-TOKEN=([^;]+)/)?.[1]; // fetch('/api/update', { // method: 'POST', // headers: { 'X-XSRF-TOKEN': token }, // body: JSON.stringify(data) // });

Fix (Java): Manual Synchronizer Token Pattern for plain servlets

If you're not using Spring Security, implement the Synchronizer Token Pattern manually. Generate a random token when the session starts, embed it in every form as a hidden field, and verify it server-side before processing any state-changing request. Reject requests where the token is missing or doesn't match.

// SAFE: Manual CSRF token generation and validation. public class CsrfUtils { private static final String CSRF_TOKEN_ATTR = "csrfToken"; /** Generate and store a CSRF token in the session. */ public static String generateToken(HttpSession session) { String token = UUID.randomUUID().toString(); session.setAttribute(CSRF_TOKEN_ATTR, token); return token; } /** Validate the submitted token against the session token. */ public static boolean isValid(HttpServletRequest req) { HttpSession session = req.getSession(false); if (session == null) return false; String sessionToken = (String) session.getAttribute(CSRF_TOKEN_ATTR); String submittedToken = req.getParameter("_csrf"); return sessionToken != null && sessionToken.equals(submittedToken); } } // In the servlet — validate before processing: @WebServlet("/update-email") public class UpdateEmailServlet extends HttpServlet { protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException { // Step 1: Reject requests with invalid or missing CSRF token. if (!CsrfUtils.isValid(req)) { resp.sendError(403, "CSRF token validation failed."); return; } // Step 2: Process the request only after validation. String userId = (String) req.getSession().getAttribute("userId"); String newEmail = req.getParameter("email"); userService.updateEmail(userId, newEmail); // Step 3: Rotate the token after use (one-time token). CsrfUtils.generateToken(req.getSession()); resp.getWriter().write("Email updated successfully."); } }

Fix (JS): Express.js with csrf-csrf middleware + SameSite cookies

Use the csrf-csrf (Double Submit Cookie) middleware for Express.js. It generates a token, stores it in a signed cookie, and validates it against a hidden form field or header on every state-changing request. Combine this with SameSite=Strict or Lax on session cookies for defense in depth.

// SAFE: CSRF protection with csrf-csrf + SameSite cookies. const express = require('express'); const session = require('express-session'); const { doubleCsrf } = require('csrf-csrf'); const cookieParser = require('cookie-parser'); const app = express(); app.use(cookieParser('signing-secret')); app.use(express.urlencoded({ extended: true })); // Session cookie with SameSite for defense in depth. app.use(session({ secret: 'session-secret', resave: false, saveUninitialized: false, cookie: { httpOnly: true, secure: true, // HTTPS only sameSite: 'lax' // Blocks cross-origin POST requests } })); // CSRF middleware — Double Submit Cookie pattern. const { doubleCsrfProtection, generateToken } = doubleCsrf({ getSecret: () => 'csrf-secret', cookieName: '__csrf', cookieOptions: { sameSite: 'lax', secure: true, httpOnly: true } }); // Provide the CSRF token to all rendered views. app.use((req, res, next) => { res.locals.csrfToken = generateToken(req, res); next(); }); // Apply CSRF protection to all state-changing routes. app.post('/change-password', doubleCsrfProtection, (req, res) => { // Token was validated by middleware — this is a legitimate request. userService.changePassword(req.session.userId, req.body.password); res.send('Password changed.'); }); // In EJS/Pug templates: // <form method="POST" action="/change-password"> // <input type="hidden" name="_csrf" value="<%= csrfToken %>" /> // <input type="password" name="password" /> // <button type="submit">Change Password</button> // </form>
