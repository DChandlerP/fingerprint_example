Why Is This an Issue?

CWE-352 flags web applications that process state-changing requests without verifying that the request was intentionally initiated by the authenticated user. In a CSRF attack, a malicious website tricks the victim's browser into sending a forged request — complete with the victim's session cookie — to the vulnerable application. Because the browser automatically attaches cookies, the server cannot distinguish legitimate requests from forged ones.

Spring Security enables CSRF protection by default. SAST tools flag this weakness when they detect csrf().disable() or csrf(csrf -> csrf.disable()) in your Spring Security configuration — or when state-changing endpoints in plain Servlets or Express lack any token verification.

The impact depends on what the targeted endpoint does. CSRF can change passwords, transfer funds, modify account settings, or elevate privileges — all without the user's knowledge.

This weakness is ranked in the SANS/CWE Top 25 and maps to OWASP A01:2021 — Broken Access Control. The primary defense is the Synchronizer Token Pattern — which Spring Security implements automatically when CSRF is not disabled.

Examples of the Issue

Dangerous: Disabling Spring Security's built-in CSRF protection

Spring Security enables CSRF protection by default. Calling csrf().disable() turns it off entirely — a common but dangerous shortcut developers take during development and forget to revert. This is the most frequent SAST finding for CWE-352 in Spring applications. Both the legacy WebSecurityConfigurerAdapter style and the modern SecurityFilterChain bean style are shown.

// DANGEROUS: Disabling Spring Security's built-in CSRF protection. // --- Legacy style (Spring Security 5.x / WebSecurityConfigurerAdapter) --- @Configuration @EnableWebSecurity public class SecurityConfig extends WebSecurityConfigurerAdapter { @Override protected void configure(HttpSecurity http) throws Exception { http .csrf().disable() // ← SAST flags this — CSRF is off globally .authorizeRequests() .antMatchers("/login", "/register").permitAll() .anyRequest().authenticated() .and() .formLogin() .loginPage("/login"); } } // --- Modern style (Spring Security 6.x / SecurityFilterChain bean) --- @Configuration @EnableWebSecurity public class SecurityConfig { @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { http .csrf(csrf -> csrf.disable()) // ← Same problem, lambda DSL .authorizeHttpRequests(auth -> auth .requestMatchers("/login", "/register").permitAll() .anyRequest().authenticated() ) .formLogin(form -> form.loginPage("/login")); return http.build(); } }

Dangerous: Spring REST API with cookie-based auth but CSRF disabled

A common misconception is that REST APIs don't need CSRF protection. This is true only if the API uses stateless authentication (e.g., Authorization: Bearer headers). If the API uses cookie-based sessions (common in SPAs served from the same origin), CSRF is still required — the browser will attach session cookies to cross-origin requests just like it does for form submissions.

// DANGEROUS: REST API uses session cookies but disables CSRF. @Configuration @EnableWebSecurity public class ApiSecurityConfig { @Bean public SecurityFilterChain apiFilterChain(HttpSecurity http) throws Exception { http .csrf(csrf -> csrf.disable()) // ← Dangerous with cookie auth! .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED) // ↑ Sessions ARE used — cookies are sent! ) .authorizeHttpRequests(auth -> auth .requestMatchers("/api/public/**").permitAll() .anyRequest().authenticated() ) .formLogin(Customizer.withDefaults()); return http.build(); } } // The SPA makes requests with credentials: // fetch('/api/transfer', { // method: 'POST', // credentials: 'include', ← Browser sends session cookie // body: JSON.stringify({ to: 'attacker', amount: 10000 }) // }); // // An attacker's page can do the same thing — the session cookie // is attached automatically. Without CSRF tokens, the server // cannot tell the difference.

Dangerous: No CSRF token verification in plain Java Servlet and Express.js

Applications not using Spring Security (plain Servlets, Express.js) have no CSRF protection unless explicitly implemented. The server trusts the session cookie alone, making every state-changing endpoint vulnerable to cross-origin form submissions.

// DANGEROUS: No CSRF token — server trusts cookie alone. // --- Java Servlet: no token verification --- @WebServlet("/update-email") public class UpdateEmailServlet extends HttpServlet { protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws IOException { HttpSession session = req.getSession(false); if (session == null) { resp.sendRedirect("/login"); return; } String userId = (String) session.getAttribute("userId"); String newEmail = req.getParameter("email"); // Attacker-controlled // No CSRF verification — cookie is the only check. userService.updateEmail(userId, newEmail); } } // --- Express.js: no CSRF protection --- app.post('/change-password', (req, res) => { if (!req.session.userId) return res.status(401).send('Not logged in.'); // No CSRF token — session cookie is the only authentication. userService.changePassword(req.session.userId, req.body.password); res.send('Password changed.'); }); // --- Attacker's page performs the attack --- // <form action="https://vulnerable-app.com/update-email" method="POST"> // <input type="hidden" name="email" value="attacker@evil.com" /> // </form> // <script>document.forms[0].submit();</script>

How to Resolve

Fix (Java): Keep Spring Security CSRF enabled — Thymeleaf, JSP, and AJAX patterns

The simplest fix is to remove csrf().disable(). Spring Security's CSRF is on by default. Thymeleaf and Spring's JSP form tag library inject the token automatically. For AJAX/SPA patterns, use CookieCsrfTokenRepository so JavaScript can read the token from a cookie and send it as a header.

// SAFE: CSRF protection enabled (the default). Remove csrf().disable(). // --- Spring Security 6.x (SecurityFilterChain bean) --- @Configuration @EnableWebSecurity public class SecurityConfig { @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { http // CSRF is ON by default — simply omit any csrf().disable() call. // For server-rendered forms (Thymeleaf/JSP), no extra config needed. .authorizeHttpRequests(auth -> auth .requestMatchers("/login", "/register").permitAll() .anyRequest().authenticated() ) .formLogin(form -> form.loginPage("/login")); return http.build(); } } // --- Thymeleaf: token is injected automatically --- // <form th:action="@{/update-email}" method="post"> // <!-- Hidden _csrf field is added automatically by Thymeleaf --> // <input type="email" name="email" /> // <button type="submit">Update</button> // </form> // --- JSP with Spring form tags: also automatic --- // <form:form action="/update-email" method="POST"> // <!-- _csrf hidden field injected by Spring form tag --> // <form:input path="email" /> // <input type="submit" value="Update" /> // </form:form> // --- Plain JSP (without form tags): include manually --- // <form method="POST" action="/update-email"> // <input type="hidden" name="${_csrf.parameterName}" value="${_csrf.token}" /> // <input type="email" name="email" /> // <button type="submit">Update</button> // </form>

Fix (Java): Spring CSRF with SPA / AJAX — CookieCsrfTokenRepository

For SPAs and AJAX-heavy applications, configure Spring to store the CSRF token in a cookie that JavaScript can read. The SPA reads the XSRF-TOKEN cookie and sends it back as the X-XSRF-TOKEN header on every state-changing request. This is the recommended pattern for React/Angular/Vue frontends backed by Spring Boot.

// SAFE: Cookie-based CSRF token for SPA / AJAX applications. @Configuration @EnableWebSecurity public class SpaSecurityConfig { @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { http .csrf(csrf -> csrf .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()) // ↑ Stores token in XSRF-TOKEN cookie (readable by JS). // Spring validates the X-XSRF-TOKEN header automatically. .csrfTokenRequestHandler(new SpaCsrfTokenRequestHandler()) ) .authorizeHttpRequests(auth -> auth .requestMatchers("/api/public/**").permitAll() .anyRequest().authenticated() ); return http.build(); } } // Spring Security 6.x requires a CsrfTokenRequestHandler for SPAs: public class SpaCsrfTokenRequestHandler extends CsrfTokenRequestAttributeHandler { private final CsrfTokenRequestHandler delegate = new XorCsrfTokenRequestAttributeHandler(); @Override public void handle(HttpServletRequest req, HttpServletResponse res, Supplier<CsrfToken> csrfToken) { this.delegate.handle(req, res, csrfToken); } @Override public String resolveCsrfTokenValue(HttpServletRequest req, CsrfToken csrfToken) { String headerValue = req.getHeader(csrfToken.getHeaderName()); return (headerValue != null) ? super.resolveCsrfTokenValue(req, csrfToken) : this.delegate.resolveCsrfTokenValue(req, csrfToken); } } // --- SPA JavaScript (React, Angular, Vue) --- // Read the XSRF-TOKEN cookie and send it as a header: // // const token = document.cookie // .split('; ') // .find(row => row.startsWith('XSRF-TOKEN=')) // ?.split('=')[1]; // // fetch('/api/transfer', { // method: 'POST', // credentials: 'include', // headers: { // 'Content-Type': 'application/json', // 'X-XSRF-TOKEN': token // ← Spring validates this header // }, // body: JSON.stringify({ to: 'alice', amount: 50 }) // }); // // Angular's HttpClient sends X-XSRF-TOKEN automatically.

When to disable CSRF, Express.js protection, and defense-in-depth

CSRF protection can be safely disabled only for truly stateless APIs that authenticate exclusively via Authorization headers (Bearer tokens, API keys) and never use cookies. For Express.js applications, use the csrf-csrf middleware. Always combine CSRF tokens with SameSite cookie attributes for defense-in-depth.

// --- When it IS safe to disable CSRF (Spring) --- // Only for purely stateless APIs with NO cookie-based auth: @Bean public SecurityFilterChain statelessApiChain(HttpSecurity http) throws Exception { http .csrf(csrf -> csrf.disable()) // OK here — no cookies used at all. .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS) // ↑ STATELESS — no session cookie created. ) .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()) // Auth via Authorization: Bearer header only. ); return http.build(); } // --- Express.js: csrf-csrf middleware --- const { doubleCsrf } = require('csrf-csrf'); const cookieParser = require('cookie-parser'); app.use(cookieParser('signing-secret')); const { doubleCsrfProtection, generateToken } = doubleCsrf({ getSecret: () => 'csrf-secret', cookieName: '__csrf', cookieOptions: { sameSite: 'lax', secure: true, httpOnly: true } }); app.use((req, res, next) => { res.locals.csrfToken = generateToken(req, res); next(); }); app.post('/change-password', doubleCsrfProtection, (req, res) => { // Token validated by middleware — legitimate request. userService.changePassword(req.session.userId, req.body.password); }); // --- Defense-in-depth: SameSite cookies --- // Set on ALL session cookies (Spring Boot application.properties): // server.servlet.session.cookie.same-site=lax // server.servlet.session.cookie.secure=true // server.servlet.session.cookie.http-only=true // // SameSite=Lax blocks cross-origin POST requests from sending cookies. // This alone prevents most CSRF, but tokens remain essential because: // 1. SameSite=Lax still allows top-level GET navigations with cookies. // 2. Older browsers may not support SameSite. // 3. Defense-in-depth is a security best practice.
