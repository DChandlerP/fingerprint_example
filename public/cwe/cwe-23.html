Why Is This an Issue?

CWE-23 flags code that constructs file paths from user-controlled input without proper validation, allowing an attacker to traverse directories with sequences like ../ to read, overwrite, or delete files outside the intended directory. This is commonly known as a directory traversal or path traversal attack.

Path traversal vulnerabilities can lead to:

  • Sensitive file disclosure — reading /etc/passwd, configuration files, or application source code
  • Credential theft — accessing .env files, database connection strings, or API keys
  • Remote code execution — overwriting application code, templates, or uploading web shells to executable directories
  • Denial of service — deleting critical files or filling disk with attacker-controlled uploads

This weakness is part of OWASP A01:2021 — Broken Access Control. The primary defense is to canonicalize the resolved path and verify it stays within the intended base directory. Never rely on stripping ../ sequences alone — there are dozens of encoding and double-encoding bypasses.

Examples of the Issue

Dangerous: File read/download with unvalidated path (Java / Servlet / Spring)

Concatenating user input directly into a file path allows an attacker to escape the intended directory. A request like GET /download?file=../../../etc/passwd reads arbitrary files from the server. Even using replace("../", "") is insufficient — attackers use ....// or URL-encoded sequences to bypass it.

// DANGEROUS: User-controlled file path — path traversal. // --- Servlet: direct file download --- @WebServlet("/download") public class DownloadServlet extends HttpServlet { private static final String UPLOAD_DIR = "/var/app/uploads/"; protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws IOException { String filename = req.getParameter("file"); // Attacker sends: ?file=../../../etc/passwd File file = new File(UPLOAD_DIR + filename); // ↑ Path traversal — no validation! resp.setContentType("application/octet-stream"); Files.copy(file.toPath(), resp.getOutputStream()); } } // --- Spring MVC: serving user-uploaded files --- @GetMapping("/files/{name}") public ResponseEntity<Resource> getFile(@PathVariable String name) { // Attacker sends: /files/..%2F..%2F..%2Fetc%2Fpasswd Path path = Paths.get("/var/app/uploads/" + name); Resource resource = new UrlResource(path.toUri()); return ResponseEntity.ok().body(resource); // ↑ No path validation! } // --- Weak sanitization that is easily bypassed --- String filename = req.getParameter("file"); filename = filename.replace("../", ""); // INSUFFICIENT! // Bypass: ....// → after replace → ../ // Bypass: ..%2F → URL decoding happens before replace // Bypass: ..\ → backslash on Windows

Dangerous: File serving and uploads with unvalidated paths (Node.js / Express)

Using path.join() or string concatenation with user input does not prevent path traversal — path.join() resolves .. segments rather than rejecting them. Express's static middleware is safe, but custom file-serving routes are a common source of this vulnerability.

// DANGEROUS: User-controlled path — arbitrary file read. const express = require('express'); const path = require('path'); const fs = require('fs'); const app = express(); const UPLOAD_DIR = '/var/app/uploads'; // Custom file-serving route — path traversal! app.get('/files/:name', (req, res) => { const filePath = path.join(UPLOAD_DIR, req.params.name); // path.join('/var/app/uploads', '../../etc/passwd') // → '/var/etc/passwd' — RESOLVES the traversal! res.sendFile(filePath); }); // File upload with user-controlled destination — path traversal! app.post('/upload', (req, res) => { const destName = req.body.filename; // Attacker controls this const destPath = path.join(UPLOAD_DIR, destName); // Attacker sends filename: ../../app/server.js // Overwrites application code! fs.writeFileSync(destPath, req.body.content); res.json({ saved: destPath }); }); // Template/view rendering with user-controlled path app.get('/page/:name', (req, res) => { res.render(req.params.name); // Attacker: /page/../../etc/passwd // Some template engines will read arbitrary files! });

Dangerous: Zip Slip — path traversal during archive extraction (Java & JavaScript)

Zip Slip is a path traversal variant where a malicious archive contains entries with names like ../../../webapps/ROOT/shell.jsp. When the archive is extracted without validating each entry's resolved path, the attacker writes files to arbitrary locations — commonly used to deploy web shells.

// DANGEROUS: Zip Slip — archive entry names not validated. // --- Java: extracting a ZIP file --- ZipInputStream zis = new ZipInputStream(new FileInputStream(uploadedZip)); ZipEntry entry; while ((entry = zis.getNextEntry()) != null) { // entry.getName() could be: ../../../opt/tomcat/webapps/ROOT/shell.jsp File outFile = new File(EXTRACT_DIR, entry.getName()); // ↑ Path traversal via archive entry name! FileOutputStream fos = new FileOutputStream(outFile); byte[] buf = new byte[1024]; int len; while ((len = zis.read(buf)) > 0) { fos.write(buf, 0, len); } fos.close(); } // --- JavaScript (Node.js): extracting with adm-zip --- const AdmZip = require('adm-zip'); const zip = new AdmZip(uploadedFile); zip.getEntries().forEach(entry => { // entry.entryName could contain: ../../config/secrets.json const outPath = path.join(extractDir, entry.entryName); fs.writeFileSync(outPath, entry.getData()); // ↑ Zip Slip — writes outside extractDir! });

How to Resolve

Fix (Java): Canonicalize and validate the resolved path stays within the base directory

Resolve the path to its canonical (absolute, normalized) form, then verify it starts with the canonical base directory. This handles all encoding tricks, symbolic links, and double-dot bypasses. Apply the same check to Zip Slip — validate every archive entry's resolved path before extraction.

// SAFE: Canonical path validation prevents all traversal bypasses. import java.io.File; import java.io.IOException; import java.nio.file.Path; public class SafeFileUtil { private static final String BASE_DIR = "/var/app/uploads"; /** * Resolves a user-supplied filename and validates it stays * within the base directory. Returns null if invalid. */ public static File safeResolve(String userFilename) throws IOException { if (userFilename == null || userFilename.isBlank()) return null; File base = new File(BASE_DIR).getCanonicalFile(); File requested = new File(base, userFilename).getCanonicalFile(); // Canonical path resolves ../, symlinks, encoding — then we check. if (!requested.toPath().startsWith(base.toPath())) { return null; // Traversal attempt — reject. } return requested; } } // --- Usage in Servlet --- @WebServlet("/download") public class SafeDownloadServlet extends HttpServlet { protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws IOException { File file = SafeFileUtil.safeResolve(req.getParameter("file")); if (file == null || !file.exists()) { resp.sendError(404, "File not found"); return; } resp.setContentType("application/octet-stream"); Files.copy(file.toPath(), resp.getOutputStream()); } } // --- Zip Slip fix: validate every entry before extraction --- ZipEntry entry; File baseDir = new File(EXTRACT_DIR).getCanonicalFile(); while ((entry = zis.getNextEntry()) != null) { File outFile = new File(baseDir, entry.getName()).getCanonicalFile(); if (!outFile.toPath().startsWith(baseDir.toPath())) { throw new SecurityException("Zip Slip detected: " + entry.getName()); } // Safe to extract — path is within the base directory. FileOutputStream fos = new FileOutputStream(outFile); // ... write file contents ... }

Fix (JS): path.resolve() + startsWith() guard and express.static()

Use path.resolve() (not path.join()) to get the absolute, normalized path, then verify it starts with the base directory. For static file serving, prefer Express's built-in express.static() middleware, which has path traversal protections built in.

// SAFE: Resolve + startsWith validation for Node.js / Express. const express = require('express'); const path = require('path'); const fs = require('fs'); const app = express(); const UPLOAD_DIR = path.resolve('/var/app/uploads'); /** * Resolves a user-supplied filename and validates it stays * within the base directory. Returns null if traversal detected. */ function safeResolvePath(baseDir, userInput) { const resolved = path.resolve(baseDir, userInput); if (!resolved.startsWith(baseDir + path.sep) && resolved !== baseDir) { return null; // Traversal attempt — reject. } return resolved; } // --- Safe file download route --- app.get('/files/:name', (req, res) => { const safePath = safeResolvePath(UPLOAD_DIR, req.params.name); if (!safePath || !fs.existsSync(safePath)) { return res.status(404).json({ error: 'File not found' }); } res.sendFile(safePath); }); // --- Safe file upload route --- app.post('/upload', (req, res) => { const safePath = safeResolvePath(UPLOAD_DIR, req.body.filename); if (!safePath) { return res.status(400).json({ error: 'Invalid filename' }); } fs.writeFileSync(safePath, req.body.content); res.json({ saved: path.basename(safePath) }); }); // --- Best practice: use express.static for serving files --- // express.static already prevents path traversal internally. app.use('/uploads', express.static(UPLOAD_DIR)); // --- Zip Slip fix (adm-zip) --- const AdmZip = require('adm-zip'); const zip = new AdmZip(uploadedFile); const extractDir = path.resolve('/var/app/extracted'); zip.getEntries().forEach(entry => { const outPath = path.resolve(extractDir, entry.entryName); if (!outPath.startsWith(extractDir + path.sep)) { throw new Error('Zip Slip detected: ' + entry.entryName); } fs.writeFileSync(outPath, entry.getData()); });

Defense-in-Depth: Indirect references, chroot, and least-privilege file access

The most robust approach avoids user-controlled file paths entirely. Use indirect references (database IDs or UUIDs) that map to file paths server-side. Combine with OS-level restrictions like chroot jails, container filesystem isolation, and least-privilege file permissions as defense-in-depth.

// SAFE: Indirect references — user never controls file paths. // --- Java: database-backed file downloads --- @WebServlet("/download") public class IndirectDownloadServlet extends HttpServlet { protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws IOException, SQLException { String fileId = req.getParameter("id"); // UUID, not a filename // Look up actual path from database — user cannot control it. FileRecord record = fileDao.findById(fileId); if (record == null || !record.belongsTo(currentUser(req))) { resp.sendError(403, "Access denied"); return; } File file = new File(record.getStoragePath()); resp.setContentType(record.getContentType()); resp.setHeader("Content-Disposition", "attachment; filename=\"" + record.getOriginalName() + "\""); Files.copy(file.toPath(), resp.getOutputStream()); } } // --- JavaScript (Express): UUID-based file access --- app.get('/files/:id', async (req, res) => { // Look up by UUID — no file path in the request. const record = await db.files.findOne({ id: req.params.id, userId: req.user.id // Authorization check! }); if (!record) return res.status(404).json({ error: 'Not found' }); res.download(record.storagePath, record.originalName); }); // --- Additional hardening --- // 1. Allowlist file extensions: // if (!['.pdf','.png','.docx'].includes(path.extname(filename))) // return res.status(400).send('Unsupported file type'); // // 2. Run app in a container with read-only filesystem // (only the upload volume is writable). // // 3. Use chroot or Linux namespaces to limit visible filesystem. // // 4. Set restrictive file permissions: // chmod 640 on uploaded files, chown to app user.
