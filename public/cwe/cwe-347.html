Why Is This an Issue?

CWE-347 flags code that accepts a JWT (JSON Web Token) without properly verifying its cryptographic signature. A JWT consists of three parts: header, payload, and signature. If the signature is not validated — or if the library is configured to accept the "none" algorithm — an attacker can forge tokens with arbitrary claims (roles, user IDs, permissions) and gain unauthorized access.

Common ways this vulnerability manifests:

  • Algorithm "none" attack — the token header specifies "alg": "none", and the library accepts it as valid without any signature
  • Algorithm confusion (RS256 → HS256) — the attacker changes the algorithm from asymmetric (RS256) to symmetric (HS256) and signs with the public key, which the server then uses as the HMAC secret
  • Missing verification — code decodes the JWT payload without calling a verification function at all
  • Key not provided — verification is called but with a null, empty, or default key

This weakness maps to OWASP A02:2021 — Cryptographic Failures. The fix requires always verifying the signature, explicitly specifying allowed algorithms (never trusting the token's header), and using a strong, properly managed signing key.

Examples of the Issue

Dangerous: Decoding JWT without signature verification (Java)

Using Jwts.parser() or JWT.decode() without providing a signing key parses the token but never validates the signature. An attacker can modify any claim (e.g., change "role": "user" to "role": "admin") and the application will accept it. Some libraries also accept "alg": "none" by default if no algorithm restriction is configured.

// DANGEROUS: Decodes JWT payload WITHOUT verifying the signature. import io.jsonwebtoken.Jwts; import io.jsonwebtoken.Claims; public class AuthFilter { public Claims extractClaims(String token) { // ❌ parserBuilder() without signingKey — signature is IGNORED. return Jwts.parserBuilder() .build() // No signing key set! .parseClaimsJwt(token) // parseClaimsJwt = unsigned JWT .getBody(); // Attacker can forge any claims — role, userId, permissions. } } // Also dangerous — using a JWT library's decode (not verify) method: import com.auth0.jwt.JWT; import com.auth0.jwt.interfaces.DecodedJWT; DecodedJWT decoded = JWT.decode(token); // Decode only — NO verification! String userId = decoded.getClaim("sub").asString(); // Attacker controls "sub" — impersonates any user. // Also dangerous — weak/default signing key: String key = "secret"; // Trivially guessable — brute-force in seconds. Claims claims = Jwts.parserBuilder() .setSigningKey(key.getBytes()) // Weak key! .build() .parseClaimsJws(token) .getBody();

Dangerous: Algorithm confusion and missing algorithm restriction (Node.js)

The popular jsonwebtoken library defaults to trusting the algorithm specified in the token's header. If the server uses RS256 (asymmetric) but doesn't restrict the allowed algorithms, an attacker can change the header to "alg": "HS256" and sign with the public key — which the server then treats as the HMAC secret. The signature validates, and the forged token is accepted.

// DANGEROUS: No algorithm restriction — vulnerable to alg confusion. const jwt = require('jsonwebtoken'); const fs = require('fs'); const publicKey = fs.readFileSync('public.pem'); // ❌ No `algorithms` option — trusts the token's "alg" header. function verifyToken(token) { return jwt.verify(token, publicKey); // Attacker sets alg=HS256 and signs with the public key. // Server uses publicKey as HMAC secret → signature matches! } // ALSO DANGEROUS: Using decode() instead of verify(). const payload = jwt.decode(token); // NO verification at all. const userId = payload.sub; // Attacker-controlled. // ALSO DANGEROUS: Empty or null secret. jwt.verify(token, '', { algorithms: ['HS256'] }); // Empty secret! jwt.verify(token, null, { algorithms: ['HS256'] }); // Null secret! // ALSO DANGEROUS: Catching verification errors and proceeding anyway. function unsafeVerify(token) { try { return jwt.verify(token, secret); } catch (err) { // ❌ Falling back to decode on verification failure! return jwt.decode(token); } }

Dangerous: Custom JWT filter that skips verification on failure (Spring)

In Spring applications, custom JWT filters sometimes catch signature verification exceptions and silently proceed with the unverified token, or use a decode-only method as a fallback. This effectively makes authentication optional — an attacker with any malformed or self-signed token passes the filter.

// DANGEROUS: Custom JWT filter that silently ignores verification failures. import io.jsonwebtoken.*; @Component public class JwtAuthFilter extends OncePerRequestFilter { @Override protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain chain) throws Exception { String header = request.getHeader("Authorization"); if (header != null && header.startsWith("Bearer ")) { String token = header.substring(7); try { Claims claims = Jwts.parserBuilder() .setSigningKey(secretKey) .build() .parseClaimsJws(token) .getBody(); setAuthentication(claims); } catch (JwtException e) { // ❌ DANGEROUS: Catches ALL JWT exceptions and proceeds. // Invalid signature, expired, malformed — all silently pass. log.warn("JWT verification failed, proceeding anyway"); } } chain.doFilter(request, response); // Unauthenticated requests reach protected endpoints! } } // Also dangerous — using the unsigned parser variant: Jws<Claims> jws = Jwts.parserBuilder().build() .parseClaimsJwt(token); // parseClaimsJwt (not Jws!) = no sig check

How to Resolve

Fix (Java): Verify with explicit algorithm and strong signing key

Always use parseClaimsJws() (not parseClaimsJwt()) which requires a valid signature. Provide a strong signing key (at least 256 bits for HMAC, 2048-bit RSA, or P-256 EC). JJWT 0.12+ enforces algorithm restrictions automatically when a key is provided — ensuring "alg": "none" is always rejected.

// SAFE: Signature verified with strong key, algorithm enforced. import io.jsonwebtoken.*; import io.jsonwebtoken.security.Keys; import javax.crypto.SecretKey; public class JwtService { // Generate a cryptographically strong key (256+ bits for HS256). // In production, load from a secrets manager (vault, env var, etc.). private static final SecretKey SIGNING_KEY = Keys.secretKeyFor(SignatureAlgorithm.HS256); /** Verify and extract claims — rejects unsigned/tampered tokens. */ public Claims verifyToken(String token) { return Jwts.parserBuilder() .setSigningKey(SIGNING_KEY) // Key MUST be set. .build() .parseClaimsJws(token) // Jws = signature required! .getBody(); // Throws SignatureException if tampered. // Throws UnsupportedJwtException if alg=none. // Throws ExpiredJwtException if exp claim is past. } } // For asymmetric (RS256/ES256) — verify with the PUBLIC key: import java.security.PublicKey; public Claims verifyWithPublicKey(String token, PublicKey publicKey) { return Jwts.parserBuilder() .setSigningKey(publicKey) // RSA/EC public key .build() .parseClaimsJws(token) // Algorithm enforced by key type. .getBody(); }

Fix (JS): Always specify algorithms array and use verify(), never decode()

Always call jwt.verify() — never jwt.decode() — for authentication decisions. Pass an explicit algorithms array to prevent algorithm confusion attacks. Use a strong secret (256+ bits for HMAC) or proper RSA/EC keys. Never catch verification errors and fall back to an unverified path.

// SAFE: Explicit algorithm restriction prevents alg confusion. const jwt = require('jsonwebtoken'); const crypto = require('crypto'); // ─── HMAC (symmetric) ─── // Generate a strong 256-bit secret (store in env var / secrets manager). const SECRET = process.env.JWT_SECRET; // At least 32 bytes. function verifyToken(token) { // algorithms array MUST be specified — blocks alg=none and confusion. return jwt.verify(token, SECRET, { algorithms: ['HS256'], // Only accept HS256 — reject all others. maxAge: '1h', // Reject tokens older than 1 hour. clockTolerance: 30, // 30s clock skew tolerance. }); // Throws JsonWebTokenError if signature is invalid. // Throws TokenExpiredError if token is expired. // Throws NotBeforeError if token is not yet valid. // NEVER catch these and fall back to decode()! } // ─── RSA (asymmetric) ─── const fs = require('fs'); const publicKey = fs.readFileSync('public.pem'); function verifyAsymmetric(token) { return jwt.verify(token, publicKey, { algorithms: ['RS256'], // CRITICAL: prevents HS256 confusion. }); } // ─── Express middleware pattern ─── function authMiddleware(req, res, next) { const authHeader = req.headers.authorization; if (!authHeader?.startsWith('Bearer ')) { return res.status(401).json({ error: 'Missing token' }); } try { req.user = verifyToken(authHeader.slice(7)); next(); } catch (err) { // ❌ Do NOT fall back to jwt.decode() here. return res.status(401).json({ error: 'Invalid token' }); } }

Fix (Spring): Use Spring Security OAuth2 Resource Server for JWT validation

Instead of writing a custom JWT filter, use Spring Security's oauth2ResourceServer() which handles signature verification, algorithm enforcement, expiration checks, and issuer validation out of the box. It supports both JWK Set URI (for OIDC providers) and local key configuration. This eliminates entire classes of JWT vulnerabilities.

// SAFE: Spring Security handles all JWT verification automatically. // application.yml — OIDC provider auto-discovery: // spring: // security: // oauth2: // resourceserver: // jwt: // issuer-uri: https://auth.example.com // # Spring fetches JWKS, validates sig, alg, exp, iss, aud. @Configuration @EnableWebSecurity public class SecurityConfig { @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { return http .authorizeHttpRequests(auth -> auth .requestMatchers("/public/**").permitAll() .anyRequest().authenticated() ) .oauth2ResourceServer(oauth2 -> oauth2 .jwt(jwt -> jwt .jwtAuthenticationConverter(jwtAuthConverter()) ) ) .build(); // Spring Security automatically: // ✅ Verifies signature against JWKS endpoint // ✅ Rejects alg=none // ✅ Validates exp, iat, nbf claims // ✅ Validates issuer (iss) claim } } // For local symmetric key (no OIDC provider): @Bean public JwtDecoder jwtDecoder() { SecretKey key = new SecretKeySpec( secretBytes, "HmacSHA256" ); NimbusJwtDecoder decoder = NimbusJwtDecoder .withSecretKey(key) .macAlgorithm(MacAlgorithm.HS256) // Explicit algorithm. .build(); // Add additional validation (issuer, audience): decoder.setJwtValidator(new DelegatingOAuth2TokenValidator<>( JwtValidators.createDefaultWithIssuer("https://auth.example.com"), new JwtClaimValidator<>("aud", aud -> aud != null && aud.contains("my-api")) )); return decoder; }
