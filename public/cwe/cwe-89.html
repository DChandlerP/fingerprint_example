Why Is This an Issue?

CWE-89 covers SQL Injection (SQLi) — a vulnerability where user-controlled input is concatenated directly into SQL queries without parameterization or proper escaping. An attacker can inject SQL syntax to alter the query's logic, bypass authentication, extract sensitive data, modify or delete records, and in some cases execute operating system commands on the database server.

SQL injection remains one of the most critical and commonly exploited web vulnerabilities. It enables:

  • Authentication bypass — injecting ' OR '1'='1 to log in without valid credentials
  • Data exfiltration — using UNION SELECT to dump tables containing passwords, PII, or financial data
  • Data manipulation — injecting UPDATE, DELETE, or INSERT statements to corrupt or destroy data
  • Privilege escalation — reading database admin credentials or modifying user roles
  • Remote code execution — leveraging database features like xp_cmdshell (SQL Server) or LOAD_FILE() / INTO OUTFILE (MySQL) to execute OS commands

Classic / In-Band SQLi

Attacker's injected query results are returned directly in the application's response. Easiest to exploit; includes UNION-based and error-based techniques.

Blind SQLi

Application does not display query results, but the attacker infers data by observing response differences (boolean-based) or time delays (time-based).

Second-Order SQLi

Malicious input is stored safely, then later incorporated into a SQL query without parameterization — the injection triggers on a subsequent request.

This weakness is part of OWASP A03:2021 — Injection and the SANS/CWE Top 25. The primary and most effective defense is parameterized queries (prepared statements) — they separate SQL structure from data, making injection structurally impossible regardless of the input.

Examples of the Issue

Dangerous: String concatenation in SQL queries (Java / JDBC / JPA)

Directly concatenating user input into SQL strings is the classic SQLi pattern. An attacker submits ' OR '1'='1' -- as the username to bypass login, or uses UNION SELECT to extract data from other tables. This applies to raw JDBC, Spring JDBC, and even JPA/Hibernate when using string-built queries.

// DANGEROUS: User input concatenated into SQL — SQL injection. // --- JDBC: classic string concatenation --- public User login(String username, String password) throws SQLException { String sql = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'"; // ↑ Attacker input: ' OR '1'='1' -- // Resulting query: SELECT * FROM users WHERE username = '' OR '1'='1' --' // Returns all users — authentication bypassed! Statement stmt = conn.createStatement(); ResultSet rs = stmt.executeQuery(sql); // ... } // --- Spring JDBC: string concatenation in JdbcTemplate --- public List<Product> search(String category) { String sql = "SELECT * FROM products WHERE category = '" + category + "'"; // Attacker: ' UNION SELECT username, password, null FROM users -- return jdbcTemplate.query(sql, productRowMapper); } // --- JPA / Hibernate: string-built JPQL --- public List<User> findByName(String name) { String jpql = "SELECT u FROM User u WHERE u.name = '" + name + "'"; // Even JPQL is injectable when built with string concatenation! return em.createQuery(jpql, User.class).getResultList(); } // --- Stored procedure call with concatenation --- String sql = "CALL get_user_orders('" + userId + "')"; // Attacker: '); DROP TABLE orders; --

Dangerous: Template literals and string concatenation in SQL (Node.js / Express)

JavaScript template literals (`${variable}`) make SQL injection easy to introduce. Whether using mysql, pg, or better-sqlite3, interpolating request parameters into query strings creates the same vulnerability. ORMs like Sequelize and Knex are safe by default, but their raw query methods are not.

// DANGEROUS: User input interpolated into SQL — SQL injection. const express = require('express'); const mysql = require('mysql2'); const app = express(); const db = mysql.createConnection({ /* config */ }); // Login bypass via template literal interpolation. app.post('/login', (req, res) => { const { username, password } = req.body; const sql = `SELECT * FROM users WHERE username = '${username}' AND password = '${password}'`; // ↑ SQL injection — template literal! // Attacker: username = "admin' --" → password check is commented out. db.query(sql, (err, results) => { if (results.length > 0) res.json({ success: true }); }); }); // Data exfiltration via search endpoint. app.get('/products', (req, res) => { const sql = "SELECT * FROM products WHERE name LIKE '%" + req.query.q + "%'"; // Attacker: q = %' UNION SELECT id, email, password FROM users -- db.query(sql, (err, results) => res.json(results)); }); // Sequelize raw query — ORM does NOT protect raw queries! const results = await sequelize.query( `SELECT * FROM orders WHERE user_id = ${req.params.id}`, { type: QueryTypes.SELECT } ); // Attacker: /orders/1; DROP TABLE orders;

Dangerous: SQL injection in ORDER BY, table names, and dynamic identifiers (Java & JavaScript)

Prepared statements cannot parameterize identifiers (column names, table names) or SQL keywords like ORDER BY direction. These are common blind spots where developers concatenate input even when using parameterized queries elsewhere. Attackers exploit ORDER BY injection for blind data extraction via conditional sorting.

// DANGEROUS: Identifiers and ORDER BY cannot use query parameters. // --- Java: ORDER BY with user-controlled column --- String sortCol = req.getParameter("sort"); // e.g. "name" String sql = "SELECT * FROM products ORDER BY " + sortCol; // Attacker: sort = (CASE WHEN (SELECT password FROM users LIMIT 1) // LIKE 'a%' THEN name ELSE price END) // Blind SQLi — response order reveals data character by character. // --- Java: dynamic table name --- String table = req.getParameter("report"); String sql = "SELECT * FROM " + table + " WHERE year = 2024"; // Attacker: report = users; DROP TABLE audit_log; -- // --- JavaScript: dynamic column in Knex --- app.get('/products', (req, res) => { const sortBy = req.query.sort || 'name'; // Knex orderBy does NOT sanitize — it's treated as a raw identifier. knex('products').orderBy(sortBy).then(rows => res.json(rows)); // Attacker: sort = (CASE WHEN 1=1 THEN name ELSE price END) }); // --- JavaScript: dynamic WHERE column --- const col = req.query.field; // "email" const val = req.query.value; // attacker-controlled const sql = `SELECT * FROM users WHERE ${col} = ?`; // ↑ Identifier — can't be parameterized! // Attacker: field = 1=1 UNION SELECT * FROM secrets WHERE 1

How to Resolve

Fix (Java): Parameterized queries with PreparedStatement, JPA, and Spring JDBC

Parameterized queries (prepared statements) are the definitive fix for SQL injection. The database driver separates the query structure from the data — user input is always treated as a literal value, never as SQL syntax. This applies to JDBC PreparedStatement, JPA named parameters, and Spring's JdbcTemplate.

// SAFE: Parameterized queries — SQL injection is structurally impossible. // --- JDBC PreparedStatement --- public User login(String username, String password) throws SQLException { String sql = "SELECT * FROM users WHERE username = ? AND password = ?"; PreparedStatement ps = conn.prepareStatement(sql); ps.setString(1, username); // Bound as data, not SQL syntax. ps.setString(2, password); // ' OR '1'='1 is treated as a literal string. ResultSet rs = ps.executeQuery(); // ... } // --- Spring JdbcTemplate with positional parameters --- public List<Product> search(String category) { String sql = "SELECT * FROM products WHERE category = ?"; return jdbcTemplate.query(sql, productRowMapper, category); // category is bound — UNION SELECT is impossible. } // --- JPA / Hibernate with named parameters --- public List<User> findByName(String name) { return em.createQuery("SELECT u FROM User u WHERE u.name = :name", User.class) .setParameter("name", name) .getResultList(); } // --- Spring Data JPA repository (parameterized by default) --- public interface UserRepository extends JpaRepository<User, Long> { List<User> findByUsername(String username); // Spring generates a parameterized query automatically. @Query("SELECT u FROM User u WHERE u.email = :email") Optional<User> findByEmail(@Param("email") String email); } // --- ORDER BY: use an allowlist, never concatenate --- private static final Set<String> ALLOWED_SORT_COLS = Set.of("name", "price", "created_at"); String sortCol = ALLOWED_SORT_COLS.contains(req.getParameter("sort")) ? req.getParameter("sort") : "name"; // Default to safe value. String sql = "SELECT * FROM products ORDER BY " + sortCol; // Column name is validated against allowlist — safe.

Fix (JS): Parameterized queries with mysql2, pg, Knex, and Sequelize

All major Node.js database libraries support parameterized queries via ? placeholders (MySQL) or $1, $2 placeholders (PostgreSQL). ORMs like Sequelize and query builders like Knex parameterize automatically when used correctly — but their raw query methods require explicit parameter binding.

// SAFE: Parameterized queries in Node.js — all major drivers. // --- mysql2: placeholder parameters --- app.post('/login', (req, res) => { const { username, password } = req.body; const sql = 'SELECT * FROM users WHERE username = ? AND password = ?'; db.query(sql, [username, password], (err, results) => { // ↑ Parameters bound separately — safe. if (results.length > 0) res.json({ success: true }); }); }); // --- pg (PostgreSQL): $1, $2 placeholders --- const { Pool } = require('pg'); const pool = new Pool(); app.get('/products', async (req, res) => { const sql = 'SELECT * FROM products WHERE name ILIKE $1'; const results = await pool.query(sql, [`%${req.query.q}%`]); res.json(results.rows); }); // --- Knex query builder (parameterized by default) --- app.get('/users/:id', async (req, res) => { const user = await knex('users').where({ id: req.params.id }).first(); // Knex generates: SELECT * FROM users WHERE id = ? with bound param. res.json(user); }); // --- Sequelize ORM (parameterized by default) --- const user = await User.findOne({ where: { username: req.body.username } // Sequelize generates parameterized query automatically. }); // --- Sequelize raw query: MUST use replacements --- const results = await sequelize.query( 'SELECT * FROM orders WHERE user_id = :userId', { replacements: { userId: req.params.id }, type: QueryTypes.SELECT } ); // --- ORDER BY: allowlist validation --- const ALLOWED_SORT = ['name', 'price', 'created_at']; const sortBy = ALLOWED_SORT.includes(req.query.sort) ? req.query.sort : 'name'; const rows = await knex('products').orderBy(sortBy); // Only allowlisted column names are used — safe.

Defense-in-Depth: Least-privilege database accounts, WAF rules, and input validation

Parameterized queries are the primary defense, but defense-in-depth reduces blast radius if a query is missed. Use least-privilege database accounts (the app account cannot DROP or ALTER), stored procedures for complex queries, input validation as a secondary layer, and WAF rules to catch common injection patterns.

// SAFE: Defense-in-depth layers beyond parameterized queries. // --- 1. Least-privilege database accounts --- // Application account should ONLY have SELECT, INSERT, UPDATE on // the tables it needs. Never use root or admin accounts. // // MySQL example: // GRANT SELECT, INSERT, UPDATE ON myapp.* TO 'app_user'@'%'; // REVOKE DROP, ALTER, CREATE, DELETE ON myapp.* FROM 'app_user'@'%'; // // Even if SQLi occurs, the attacker cannot DROP TABLE or read // system tables. // --- 2. Input validation as a secondary layer (Java) --- // Validate expected format BEFORE the query — reduces attack surface. public void validateUserId(String userId) { if (!userId.matches("^[0-9]{1,10}$")) { throw new IllegalArgumentException("Invalid user ID format"); } // Even though we'll use a PreparedStatement, validating input // provides a second layer of protection. } // --- 3. Input validation (JavaScript) --- function validateProductId(id) { const parsed = parseInt(id, 10); if (isNaN(parsed) || parsed < 1) { throw new Error('Invalid product ID'); } return parsed; } // --- 4. Error handling — never expose SQL errors to users --- // DANGEROUS: res.status(500).send(err.message); // → Reveals table names, column names, DB version. // SAFE: app.use((err, req, res, next) => { console.error('DB error:', err); // Log full error internally. res.status(500).json({ error: 'An internal error occurred.' }); // Generic message — no SQL details leaked. }); // --- 5. WAF / middleware SQL injection detection --- // Cloud Armor, AWS WAF, or ModSecurity rules can catch common // patterns like: ' OR 1=1, UNION SELECT, DROP TABLE, etc. // These are a safety net, NOT a primary defense. // // Example Cloud Armor rule: // evaluatePreconfiguredExpr('sqli-v33-stable')
