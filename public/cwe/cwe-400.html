Why Is This an Issue?

CWE-400 (specifically Regular Expression Denial of Service or ReDoS) occurs when a Regular Expression is applied to a long string in a way that causes the regex engine to take an exponential amount of time to finish.

This is caused by catastrophic backtracking. When a regex contains nested quantifiers or overlapping groups (like (a+)+$) and is matched against a non-matching string, the engine tries every possible combination of splits before finally failing.

  • CPU Exhaustion — A single malicious request can peg a CPU core at 100% for minutes or even hours.
  • Service Outage — Because many web servers (like Node.js) are single-threaded or have a limited pool of workers, a few ReDoS requests can completely freeze the entire application.
  • Low Cost, High Impact — Unlike a volumetric DDoS, a ReDoS attack requires very little bandwidth from the attacker.

SAST tools flag complex regex patterns that contain "evil" structures like nested repetitions or ambiguous alternations. The primary defense is simplifying the regex, using timeout mechanisms, or choosing a non-backtracking regex engine.

Examples of the Issue

Dangerous: Evil Regex pattern (JavaScript / Node.js)

This regex for validating a comma-separated list is "evil" because of the nested + inside the *.

// DANGEROUS: Nested quantifiers lead to exponential backtracking. const regex = /^([a-zA-Z0-9 ]+,)*[a-zA-Z0-9 ]+$/; app.post('/validate', (req, res) => { const input = req.body.list; // Attacker input: "a,a,a,a,a,a,a,a,a,a,a,a,a,a,a,a,a,a,a,a,a,!" // The engine tries to split the 'a's between the two groups. // For 20 'a's, this takes millions of steps. if (regex.test(input)) { res.send("Valid"); } else { res.send("Invalid"); } });

Dangerous: Ambiguous Alternation (Java)

Patterns like (a|a)+$ are ambiguous because a single character can match multiple parts of the regex. This leads to the same backtracking death spiral.

// DANGEROUS: Ambiguous groups. Pattern p = Pattern.compile("(a|aa)+$"); Matcher m = p.matcher("aaaaaaaaaaaaaaaaaaaaaaaaaaaa!"); // Long string of 'a's // m.matches() will hang the thread. if (m.matches()) { // ... }

How to Resolve

Fix: Simplify the Regex (Flattening)

Most ReDoS-vulnerable regexes can be rewritten to be linear. Avoid nesting quantifiers and ensure that each part of an alternation matches a distinct, non-overlapping set of characters.

// SAFE: Linear regex for comma-separated list. // Instead of /^([a-z]+,)*[a-z]+$/ // Use a simpler approach: const safeRegex = /^[a-z,]+$/; // Or even better, split and validate individual items: const items = input.split(','); const allValid = items.every(item => /^[a-z]+$/.test(item.trim()));

Fix: Enforce Regex Timeouts

If you must use complex regex, wrap the execution in a timeout. Some languages (.NET) support this natively, while in others (Java/JS) you may need a wrapper or a specialized engine.

// SAFE (Conceptual): Limit execution time. // Node.js doesn't have a built-in regex timeout. // Use an external library like 'safe-regex' to detect evil patterns // or 'vm2' to run regex in a separate, timed context. // Java doesn't have a direct timeout for Matcher.matches(), // but you can implement a CharSequence wrapper that checks for // thread interruption or elapsed time.

Fix: Use a Non-Backtracking Engine (RE2)

Google's RE2 engine uses an automata-based approach that is guaranteed to run in linear time. It does not support features like backreferences (which require backtracking), making it immune to ReDoS.

// SAFE: Using re2 in Node.js instead of native RegExp. const RE2 = require('re2'); const regex = new RE2('^([a-z]+,)*[a-z]+$'); // Even with an 'evil' pattern, RE2 will finish in linear time. regex.test(maliciousInput); // Similar libraries exist for Java: com.google.re2j
