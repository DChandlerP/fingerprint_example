Why Is This an Issue?

CWE-918 covers Server-Side Request Forgery (SSRF) — a vulnerability where an attacker can induce the server-side application to make requests to an arbitrary domain or internal resource. Because the request originates from the trusted server, it can bypass firewalls, VPC restrictions, and ACLs to access internal services that are not exposed to the public internet.

SSRF is particularly critical in cloud and microservices environments because it enables:

  • Cloud Metadata Exfiltration — stealing temporary IAM credentials or instance identity by querying metadata services (e.g., 169.254.169.254)
  • Internal Port Scanning — probing the internal network to identify databases, cache servers (Redis), and administrative consoles
  • Data Extraction — reading internal configuration files or interacting with private APIs via the file:// or gopher:// protocols
  • Secondary Attacks — leveraging domestic protocols to exploit internal vulnerabilities like unauthenticated database commands

This weakness is part of OWASP A10:2021 — Server-Side Request Forgery. The primary defense is strict allowlisting of domains and IP addresses. Generic blacklisting is generally insufficient due to URL encoding tricks, DNS rebinding, and alternative IP representations (e.g., decimal, octal, or IPv6).

Examples of the Issue

Dangerous: Unvalidated URL in HTTP Client (Java / Spring RestTemplate)

When an application allows a user to specify a URL (e.g., for fetching a profile picture, importing data, or a webhook), and passes that URL directly to an HTTP client like RestTemplate or HttpClient, it creates an SSRF sink. An attacker can supply internal URLs to probe the infrastructure.

// DANGEROUS: User-controlled URL parameter passed to HTTP client. @RestController public class ProxyController { @Autowired private RestTemplate restTemplate; @GetMapping("/proxy") public byte[] fetchUrl(@RequestParam String imageUrl) { // Attacker input: http://169.254.169.254/latest/meta-data/iam/security-credentials/ // Attacker input: http://localhost:8080/admin/delete-all // Attacker input: http://10.0.0.5:6379 (Redis) return restTemplate.getForObject(imageUrl, byte[].class); // ↑ SSRF Sink: Arbitrary internal request } } // --- Using Apache HttpClient or Java 11+ HttpClient --- HttpRequest request = HttpRequest.newBuilder() .uri(URI.create(userInputUrl)) // Still vulnerable! .build();

Dangerous: Unvalidated URL in Node.js (Axios / Fetch)

Node.js applications fetching external resources are equally vulnerable. Attackers can exploit SSRF to access AWS/GCP metadata, bypass local authentication on internal admin tools, or even perform port scanning on the internal VPC network.

// DANGEROUS: User-controlled URL passed to Axios. const express = require('express'); const axios = require('axios'); const app = express(); app.get('/fetch-profile', async (req, res) => { const url = req.query.url; // Attacker: ?url=http://169.254.169.254/latest/meta-data/ // Attacker: ?url=http://127.0.0.1:2375 (Docker API) try { const response = await axios.get(url); res.send(response.data); } catch (error) { res.status(500).send('Error fetching resource'); } }); // --- Also applies to native fetch() or got --- const response = await fetch(req.query.target); // SSRF vulnerability

Dangerous: SSRF via HTTP Redirects and Weak Denylists

Simple denylists (e.g., checking if the host is "127.0.0.1") are easily bypassed. Attackers use HTTP Redirects (telling the server to follow a 302 to an internal IP), DNS Rebinding (changing the IP of a safe domain to a private IP mid-request), or alternative IP representations (e.g., 2130706433 instead of 127.0.0.1).

// DANGEROUS: Weak SSRF validation (easily bypassed). // --- Java: Prefix check bypass --- if (url.startsWith("https://trusted.com")) { restTemplate.getForObject(url, String.class); } // Bypass: https://trusted.com@evil.com (UserInfo trick) // Bypass: https://trusted.com.evil.com (Subdomain trick) // --- Node.js: Localhost check bypass --- if (new URL(url).hostname !== 'localhost') { axios.get(url); } // Bypass: http://[::1]/ (IPv6) // Bypass: http://127.0.0.1.nip.io/ (DNS Wildcard) // Bypass: Redirecting from a remote server to http://localhost

How to Resolve

Fix (Java): Domain Allowlisting and Protocol Enforcement

The most effective defense is a Strict Allowlist. Do not allow arbitrary domains; only allow requests to specific, trusted targets. Always ensure the protocol is restricted to https and disable redirect following if the client allows it.

// SAFE: Strict allowlist and protocol enforcement. private final Set<String> ALLOWED_DOMAINS = Set.of("images.trusted.com", "cdn.partner.io"); public byte[] safeFetch(String userUrl) { URI uri = URI.create(userUrl); // 1. Enforce HTTPS protocol if (!"https".equals(uri.getScheme())) { throw new SecurityException("Incorrect protocol"); } // 2. Validate against strict allowlist of domains if (!ALLOWED_DOMAINS.contains(uri.getHost())) { throw new SecurityException("Target not allowed"); } // 3. Disable Redirects in your HTTP Client config // (Example using Apache HttpClient) HttpClient client = HttpClientBuilder.create() .disableRedirectHandling() .build(); return restTemplate.getForObject(uri, byte[].class); }

Fix (JS): Resolve IP and block Private/Local Ranges

If the application requires fetching from many domains, you must manually resolve the IP address and check it against a blocklist of private network ranges (RFC 1918). This prevents users from hitting internal services even if they use DNS rebinding or URL encoding tricks.

// SAFE: Manually resolving and checking for private IP ranges. const ipaddr = require('ipaddr.js'); const dns = require('dns').promises; async function isSafeUrl(targetUrl) { const url = new URL(targetUrl); if (url.protocol !== 'https:') return false; // Resolve domain to IP to prevent DNS Rebinding const { address } = await dns.lookup(url.hostname); const ip = ipaddr.parse(address); const range = ip.range(); // Block private, loopback, and link-local ranges const forbidden = ['private', 'loopback', 'linkLocal', 'unspecified']; return !forbidden.includes(range); } app.get('/fetch', async (req, res) => { if (!await isSafeUrl(req.query.url)) { return res.status(403).send('Forbidden target'); } const response = await axios.get(req.query.url, { maxRedirects: 0 // Disable redirects to prevent internal hop }); res.send(response.data); });

Defense-in-Depth: Cloud Metadata Protection and Network Isolation

Application-level fixes should be complemented by infrastructure security. Enforce AWS IMDSv2 (requires a session token header), which makes it nearly impossible to hit metadata via simple SSRF. Use Network Egress Filters (e.g., ISTIO service mesh or VPC egress rules) to block application servers from initiating requests to any IP range they don't explicitly need.

// INFRASTRUCTURE HARDENING: // 1. AWS IMDSv2: Enforce "Require tokens" on all EC2 instances. // Simple GET requests from an SSRF sink will fail to get metadata. // 2. VPC Egress Rules: // Restrict application security groups so they cannot initiate // outbound traffic to 169.254.169.254 or internal DB tiers. // 3. Use an Egress Proxy: // Route all application outbound traffic through a central proxy // (like Squid or Broadcom Proxy) that enforces a strict domain allowlist. // 4. Disable domestic protocols: // In Java, specifically restrict allowed protocols: // System.setProperty("sun.net.http.allowRestrictedHeaders", "false");
