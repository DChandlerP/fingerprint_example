Why Is This an Issue?

CWE-330 flags code that uses a non-cryptographic random number generator (e.g., Math.random(), java.util.Random, random.random()) in a context where unpredictability matters for security. These generators are fast and produce statistically uniform output, but their sequences are predictable — an attacker who observes a few outputs can reconstruct the internal state and predict all future values.

This weakness maps to OWASP A02:2021 — Cryptographic Failures. The fix depends entirely on context: replace the weak RNG with a cryptographically secure one (SecureRandom, crypto.randomBytes(), secrets) — but only when the use case demands it.

SAST False Positive Guidance — Context Matters

SAST tools flag every use of non-cryptographic RNGs, but not every use is a security issue. The key question is: "Could an attacker gain any advantage by predicting this value?" If the answer is no, the finding is a false positive.

🔴 True Positive — Requires CSPRNG

  • Session tokens / session IDs
  • CSRF tokens
  • Password reset links / OTP codes
  • API keys and secret generation
  • Cryptographic nonces / IVs
  • OAuth state parameters
  • Email verification tokens
  • Any value an attacker could exploit if predicted

🟢 Likely False Positive — CSPRNG Not Needed

  • UI animations and visual effects
  • Randomizing display order (e.g., carousel)
  • A/B test group assignment (non-security)
  • Load balancing / jitter timers
  • Shuffling quiz questions
  • Random placeholder text or colors
  • Monte Carlo simulations / math modeling
  • Test data and fixture generation

Tip: When triaging, search the surrounding code for keywords like token, session, password, secret, nonce, or csrf. If none of these are present and the value is only used for display or non-security logic, mark the finding as a false positive with a justification.

Examples of the Issue

Dangerous: java.util.Random for session tokens and CSRF tokens (Java)

java.util.Random uses a 48-bit linear congruential generator (LCG). Given just two consecutive outputs, an attacker can recover the internal seed and predict every future token. When used for session IDs or CSRF tokens, this allows session hijacking and cross-site request forgery. The ThreadLocalRandom variant has the same predictability problem.

// DANGEROUS: java.util.Random is predictable — never use for security tokens. import java.util.Random; public class TokenService { // LCG with 48-bit state — output is predictable from 2 samples. private static final Random rng = new Random(); /** Generate a session token. */ public String generateSessionToken() { // Attacker can predict future tokens after observing ≤2 values. StringBuilder token = new StringBuilder(); for (int i = 0; i < 32; i++) { token.append(Integer.toHexString(rng.nextInt(16))); } return token.toString(); } /** Generate a CSRF token. */ public String generateCsrfToken() { // Same predictable RNG — CSRF protection is effectively bypassed. return Long.toHexString(rng.nextLong()); } } // Also dangerous — ThreadLocalRandom is fast but equally predictable: long token = ThreadLocalRandom.current().nextLong(); // Don't use for security!

Dangerous: Math.random() for password reset tokens and OTP codes (Node.js)

Math.random() uses the V8 engine's xorshift128+ algorithm, which produces output that can be reversed to recover the internal state from a handful of observed values. When used for password reset links or one-time codes, an attacker who can trigger and observe a few resets can predict the token for any target account.

// DANGEROUS: Math.random() is predictable — state recoverable from ~5 outputs. /** Generate a 6-digit OTP code. */ function generateOTP() { // xorshift128+ can be reversed — attacker predicts future OTPs. return Math.floor(100000 + Math.random() * 900000).toString(); } /** Generate a password reset token. */ function generateResetToken() { // Token is predictable after observing a few resets. let token = ''; const chars = 'abcdefghijklmnopqrstuvwxyz0123456789'; for (let i = 0; i < 32; i++) { token += chars.charAt(Math.floor(Math.random() * chars.length)); } return token; } /** Generate an API key for a new user. */ function generateApiKey() { // API key derived from Math.random — can be predicted and forged. return 'ak_' + Math.random().toString(36).substring(2) + Math.random().toString(36).substring(2); }

Dangerous: random module for secrets and cryptographic nonces (Python)

Python's random module uses the Mersenne Twister (MT19937), which is completely reversible after observing 624 outputs. The module's own documentation explicitly warns: "should not be used for security purposes." When used for tokens, nonces, or invitation codes with access implications, the system is vulnerable to prediction attacks.

# DANGEROUS: random module uses Mersenne Twister — fully reversible. import random import string def generate_invite_code(): """Generate an invitation code that grants account access.""" # MT19937: internal state recoverable after 624 observed outputs. chars = string.ascii_letters + string.digits return ''.join(random.choice(chars) for _ in range(24)) def generate_nonce(): """Generate a nonce for a cryptographic protocol.""" # Predictable nonce = broken protocol (e.g., AES-GCM nonce reuse). return random.getrandbits(128) def generate_temp_password(): """Generate a temporary password for new accounts.""" # Attacker can predict passwords issued to other users. chars = string.ascii_letters + string.digits + '!@#$%' return ''.join(random.choices(chars, k=12))

How to Resolve

Fix (Java): Replace java.util.Random with SecureRandom

java.security.SecureRandom reads from the OS entropy pool (/dev/urandom on Linux, CryptGenRandom on Windows). Its output is computationally indistinguishable from true randomness. It is a drop-in replacement for Random — both implement the same interface.

// SAFE: SecureRandom is backed by OS entropy — output is unpredictable. import java.security.SecureRandom; import java.util.HexFormat; public class TokenService { // SecureRandom is thread-safe and seeds from /dev/urandom. private static final SecureRandom rng = new SecureRandom(); /** Generate a cryptographically secure session token. */ public String generateSessionToken() { byte[] bytes = new byte[32]; // 256 bits of entropy rng.nextBytes(bytes); return HexFormat.of().formatHex(bytes); } /** Generate a cryptographically secure CSRF token. */ public String generateCsrfToken() { byte[] bytes = new byte[32]; rng.nextBytes(bytes); return HexFormat.of().formatHex(bytes); } } // Note: java.util.Random is FINE for non-security contexts: // - Shuffling UI elements // - Random delays / jitter // - Test data generation // - Game mechanics // Only replace with SecureRandom when the value is security-sensitive.

Fix (JS): crypto.randomBytes / crypto.randomUUID for security-sensitive values

Use Node.js crypto.randomBytes() or the Web Crypto API crypto.getRandomValues() for any value that must be unpredictable. For simple unique identifiers, crypto.randomUUID() provides a cryptographically random UUID v4 with zero configuration.

// SAFE: crypto module uses OS-level CSPRNG. const crypto = require('crypto'); /** Generate a 6-digit OTP code (cryptographically random). */ function generateOTP() { // randomInt uses rejection sampling for uniform distribution. return crypto.randomInt(100000, 999999).toString(); } /** Generate a password reset token (256 bits of entropy). */ function generateResetToken() { return crypto.randomBytes(32).toString('hex'); } /** Generate an API key. */ function generateApiKey() { return 'ak_' + crypto.randomBytes(24).toString('base64url'); } /** Generate a UUID v4 (122 bits of randomness). */ function generateId() { return crypto.randomUUID(); // Built-in since Node 19+ } // Browser equivalent (Web Crypto API): // const array = new Uint8Array(32); // crypto.getRandomValues(array); // Note: Math.random() is FINE for non-security contexts: // - Randomizing UI animations or transitions // - Shuffling display order of items // - A/B test group assignment (non-auth) // - Adding jitter to retry timers // Only switch to crypto when predictability = security risk.

Fix (Python): Use the secrets module (or os.urandom) for security-sensitive values

Python 3.6+ provides the secrets module specifically for generating cryptographically strong random values. It wraps os.urandom() with convenient helpers for tokens, URLs, and passwords. For older codebases, os.urandom() or random.SystemRandom() provide the same entropy source.

# SAFE: secrets module is purpose-built for security tokens. import secrets import string def generate_invite_code(): """Generate a cryptographically secure invitation code.""" return secrets.token_urlsafe(24) # 192 bits, URL-safe base64 def generate_nonce(): """Generate a cryptographically secure 128-bit nonce.""" return secrets.token_bytes(16) # 128 bits from os.urandom def generate_temp_password(): """Generate a secure temporary password.""" chars = string.ascii_letters + string.digits + '!@#$%' return ''.join(secrets.choice(chars) for _ in range(12)) # Other useful secrets functions: secrets.token_hex(32) # 64-char hex string (256 bits) secrets.token_urlsafe(32) # URL-safe base64 string secrets.randbelow(1000000) # Secure random int below N (for OTPs) secrets.compare_digest(a,b) # Constant-time comparison (anti-timing) # Note: random module is FINE for non-security contexts: # - random.shuffle() for quiz question order # - random.choice() for placeholder data # - random.gauss() for statistical simulations # - numpy.random for ML / scientific computing # Only switch to secrets when the value protects something.
