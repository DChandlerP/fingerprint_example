Why Is This an Issue?

CWE-315 flags code that stores sensitive information — such as session tokens, passwords, personal data, or authentication credentials — in a cookie without encrypting it first. Cookies are transmitted in HTTP headers and can be read by anyone who can intercept the traffic (e.g., via a man-in-the-middle attack on a non-HTTPS connection), inspect browser storage, or exploit a cross-site scripting (XSS) vulnerability.

Storing sensitive data in cleartext cookies violates the principle of defense-in-depth. Even when HTTPS is used, cookies can be leaked through browser extensions, referrer headers, or logs. The safest approach is to never store sensitive data directly in a cookie — instead, store a random, opaque session identifier and keep the sensitive data server-side.

Examples of the Issue

Dangerous: Storing a username and role in a cleartext cookie

The cookie value is completely readable. An attacker who intercepts the traffic or accesses the browser can read the user's identity and role. Worse, they can tamper with the value (e.g., change role=user to role=admin) to escalate privileges.

// DANGEROUS: Sensitive user data stored directly in a cleartext cookie. Cookie userCookie = new Cookie("user_info", "username=jdoe&role=admin"); userCookie.setMaxAge(86400); response.addCookie(userCookie);

Dangerous: Storing an API token in a cookie without Secure or HttpOnly flags

Without the Secure flag, the cookie is sent over HTTP (unencrypted). Without HttpOnly, JavaScript (including XSS payloads) can read the token. This gives an attacker easy access to a valid API credential.

// DANGEROUS: API token in a cookie with no protection flags. Cookie tokenCookie = new Cookie("api_token", apiToken); // Missing: tokenCookie.setSecure(true); // Missing: tokenCookie.setHttpOnly(true); response.addCookie(tokenCookie);

Dangerous: Storing PII (email) directly in a cookie

Personally identifiable information should never be stored directly in a cookie. If the cookie is logged by a proxy, CDN, or WAF, the user's email is exposed in cleartext in server logs — a privacy and compliance violation (GDPR, CCPA, etc.).

// DANGEROUS: PII stored directly in a cookie value. String email = userService.getEmail(userId); Cookie prefsCookie = new Cookie("user_prefs", "email=" + email + "&theme=dark"); response.addCookie(prefsCookie);

How to Resolve

Fix: Use server-side sessions instead of cookie-stored data

The gold standard is to store only an opaque, random session ID in the cookie and keep all sensitive data server-side (in a database, Redis, or in-memory store). The session ID itself carries no meaningful information if intercepted.

// SAFE: Store sensitive data in the server-side session, not the cookie. HttpSession session = request.getSession(true); session.setAttribute("username", "jdoe"); session.setAttribute("role", "admin"); // The JSESSIONID cookie contains only a random, opaque identifier. // No sensitive data is transmitted in the cookie itself.

Fix: Always set Secure, HttpOnly, and SameSite flags

Even when storing only a session ID, always harden the cookie with proper security flags. Secure ensures HTTPS-only transmission. HttpOnly prevents JavaScript access (mitigating XSS theft). SameSite=Strict prevents cross-site request forgery (CSRF).

// SAFE: Cookie hardened with all recommended security flags. Cookie sessionCookie = new Cookie("SESSIONID", sessionId); sessionCookie.setSecure(true); // Only sent over HTTPS sessionCookie.setHttpOnly(true); // Not accessible via JavaScript sessionCookie.setMaxAge(3600); // 1-hour expiry sessionCookie.setPath("/"); // For SameSite (not directly in javax.servlet.http.Cookie API), // use the Set-Cookie header directly or Spring's ResponseCookie: // ResponseCookie cookie = ResponseCookie.from("SESSIONID", sessionId) // .secure(true).httpOnly(true).sameSite("Strict").build(); // response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString()); response.addCookie(sessionCookie);

Fix: Encrypt and sign the cookie value if client-side storage is required

If you absolutely must store data in a cookie (e.g., a stateless architecture), encrypt it with AES-GCM and sign it with an HMAC. This ensures confidentiality (can't read it) and integrity (can't tamper with it). Use a server-managed key that is rotated regularly.

// SAFE: Encrypt sensitive data before placing it in a cookie. // This example uses AES-GCM for authenticated encryption. SecretKey key = getEncryptionKey(); // Load from secure key store Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding"); byte[] iv = new byte[12]; SecureRandom.getInstanceStrong().nextBytes(iv); cipher.init(Cipher.ENCRYPT_MODE, key, new GCMParameterSpec(128, iv)); String plaintext = "username=jdoe&role=admin"; byte[] ciphertext = cipher.doFinal(plaintext.getBytes(StandardCharsets.UTF_8)); // Combine IV + ciphertext, then Base64-encode for the cookie value byte[] combined = ByteBuffer.allocate(iv.length + ciphertext.length) .put(iv).put(ciphertext).array(); String cookieValue = Base64.getUrlEncoder().withoutPadding() .encodeToString(combined); Cookie secureCookie = new Cookie("session_data", cookieValue); secureCookie.setSecure(true); secureCookie.setHttpOnly(true); response.addCookie(secureCookie);
