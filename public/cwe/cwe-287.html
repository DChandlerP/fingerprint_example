Why Is This an Issue?

CWE-287 flags code where the application fails to sufficiently verify that a user is who they claim to be, or allows access to resources without proper identity checks. This is a broad weakness that encompasses missing authentication on endpoints, broken authentication logic, and Broken Object Level Authorization (BOLA) — where a user can access or modify another user's data simply by changing an ID in the request.

Common manifestations include:

  • Missing authentication — sensitive endpoints are accessible without any login or token
  • BOLA / IDOR — authenticated users can access other users' resources by manipulating IDs in URLs, request bodies, or query parameters (e.g., GET /api/users/1234/records → change to /api/users/5678/records)
  • Broken session management — sessions that don't expire, can be fixated, or are shared across users
  • Authentication bypass — logic flaws that allow skipping authentication checks (e.g., checking a flag the client controls)

This weakness maps to OWASP A07:2021 — Identification and Authentication Failures and appears in the SANS/CWE Top 25. BOLA is also consistently the #1 API security risk in the OWASP API Security Top 10. The fix requires enforcing authentication on all sensitive endpoints and verifying that the authenticated user is authorized to access the specific resource they're requesting.

Examples of the Issue

Dangerous: BOLA — Spring controller trusts client-supplied user ID (Java/Spring)

The most common BOLA pattern: the controller accepts a user ID from the URL path or request body and returns that user's data without verifying that the authenticated user owns the resource. An attacker simply iterates through IDs to exfiltrate every user's records. This is OWASP API Security #1 — the single most exploited API vulnerability.

// DANGEROUS: BOLA — any authenticated user can access any other user's data. @RestController @RequestMapping("/api/users") public class UserController { @GetMapping("/{userId}/profile") public ResponseEntity<UserProfile> getProfile(@PathVariable Long userId) { // ❌ No ownership check — trusts the userId from the URL. // Attacker changes /api/users/1234/profile → /api/users/5678/profile UserProfile profile = userService.findById(userId); return ResponseEntity.ok(profile); } @GetMapping("/{userId}/documents") public ResponseEntity<List<Document>> getDocuments(@PathVariable Long userId) { // ❌ Same problem — any user can download any user's documents. return ResponseEntity.ok(documentService.findByUserId(userId)); } @PutMapping("/{userId}/email") public ResponseEntity<Void> updateEmail( @PathVariable Long userId, @RequestBody EmailUpdateRequest request) { // ❌ Attacker can change OTHER users' email addresses. userService.updateEmail(userId, request.getNewEmail()); return ResponseEntity.ok().build(); } }

Dangerous: Overly permissive Spring Security config and missing auth (Java/Spring)

Misconfigured Spring Security can silently expose sensitive endpoints. Using permitAll() too broadly, disabling CSRF on state-changing endpoints, or relying on URL patterns that don't match all request variants (e.g., trailing slashes, case variations) leaves gaps that attackers exploit.

// DANGEROUS: Overly permissive security configuration. @Configuration @EnableWebSecurity public class SecurityConfig { @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { return http .csrf(csrf -> csrf.disable()) // ❌ CSRF disabled globally! .authorizeHttpRequests(auth -> auth .requestMatchers("/api/public/**").permitAll() .requestMatchers("/api/**").permitAll() // ❌ Too broad — ALL API // endpoints are public! .anyRequest().authenticated() ) .build(); } } // Also dangerous — relying on security by obscurity: @RestController public class AdminController { // No @PreAuthorize, no role check — anyone with the URL can access. @GetMapping("/api/internal/admin/users") public List<User> getAllUsers() { return userService.findAll(); // ❌ "Hidden" endpoint, but no auth. } @DeleteMapping("/api/internal/admin/users/{id}") public void deleteUser(@PathVariable Long id) { userService.delete(id); // ❌ No auth, no authorization. } }

Dangerous: BOLA and missing authentication middleware (Node.js / Express)

In Express applications, forgetting to apply authentication middleware to a route — or applying it but trusting a user-supplied ID without ownership verification — creates the same BOLA vulnerability. This is especially common when routes are added incrementally and new endpoints skip the middleware chain.

// DANGEROUS: No auth middleware on sensitive routes + BOLA. const express = require('express'); const app = express(); // ❌ No authentication middleware — anyone can call this. app.get('/api/users/:userId/orders', async (req, res) => { const orders = await db.getOrders(req.params.userId); res.json(orders); // Any visitor can enumerate all users' orders. }); // Even WITH auth middleware, BOLA if you trust the URL param: const authMiddleware = require('./middleware/auth'); app.get('/api/users/:userId/settings', authMiddleware, // ✅ User is authenticated... async (req, res) => { // ❌ ...but no ownership check. req.user.id might be 1234, // while req.params.userId is 5678. Still returns 5678's data. const settings = await db.getUserSettings(req.params.userId); res.json(settings); } ); // ❌ Same pattern with POST — attacker modifies another user's data: app.post('/api/users/:userId/address', authMiddleware, async (req, res) => { // Trusts :userId from URL, ignores req.user.id. await db.updateAddress(req.params.userId, req.body); res.json({ success: true }); } );

How to Resolve

Fix (Spring): Enforce ownership — use the authenticated principal, not the URL

The core BOLA fix: derive the user identity from the authenticated session/token, not from the request. Use Spring Security's @AuthenticationPrincipal to inject the verified user, and either scope queries to that user's ID or explicitly verify ownership before returning data. For admin-level access, use @PreAuthorize role checks.

// SAFE: User identity comes from the authenticated token, not the URL. @RestController @RequestMapping("/api/users") public class UserController { // ─── Pattern 1: Remove userId from URL entirely ─── @GetMapping("/me/profile") public ResponseEntity<UserProfile> getMyProfile( @AuthenticationPrincipal UserDetails principal) { // User ID from the verified JWT/session — not the request. UserProfile profile = userService.findByUsername( principal.getUsername() ); return ResponseEntity.ok(profile); } // ─── Pattern 2: Keep userId in URL but verify ownership ─── @GetMapping("/{userId}/documents") @PreAuthorize("#userId == authentication.principal.id or hasRole('ADMIN')") public ResponseEntity<List<Document>> getDocuments( @PathVariable Long userId) { // SpEL expression ensures userId matches the authenticated user, // OR the caller has ADMIN role. Rejects mismatched IDs with 403. return ResponseEntity.ok(documentService.findByUserId(userId)); } // ─── Pattern 3: Service-layer ownership check ─── @PutMapping("/{userId}/email") public ResponseEntity<Void> updateEmail( @PathVariable Long userId, @RequestBody EmailUpdateRequest request, @AuthenticationPrincipal CustomUser principal) { if (!principal.getId().equals(userId)) { throw new AccessDeniedException("Cannot modify another user"); } userService.updateEmail(userId, request.getNewEmail()); return ResponseEntity.ok().build(); } }

Fix (Spring): Deny-by-default security configuration with method security

Configure Spring Security with a deny-by-default posture: require authentication on all endpoints and explicitly whitelist only truly public ones. Enable @EnableMethodSecurity for fine-grained @PreAuthorize checks at the controller/service level. This ensures new endpoints are protected by default.

// SAFE: Deny-by-default — new endpoints are protected automatically. @Configuration @EnableWebSecurity @EnableMethodSecurity // Enables @PreAuthorize, @Secured, etc. public class SecurityConfig { @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { return http .csrf(Customizer.withDefaults()) // ✅ CSRF enabled .authorizeHttpRequests(auth -> auth // Whitelist ONLY truly public endpoints: .requestMatchers("/api/public/**").permitAll() .requestMatchers("/health", "/info").permitAll() // Everything else requires authentication: .anyRequest().authenticated() // ✅ Deny-by-default ) .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults())) .sessionManagement(s -> s .sessionCreationPolicy(SessionCreationPolicy.STATELESS) ) .build(); } } // Controller with method-level authorization: @RestController @RequestMapping("/api/admin") @PreAuthorize("hasRole('ADMIN')") // ✅ Class-level role check public class AdminController { @GetMapping("/users") public List<User> getAllUsers() { return userService.findAll(); // Only ADMIN can access. } @DeleteMapping("/users/{id}") @PreAuthorize("hasRole('SUPER_ADMIN')") // ✅ Elevated privilege public void deleteUser(@PathVariable Long id) { userService.delete(id); } }

Fix (JS): Auth middleware + ownership verification on every route (Express)

Apply authentication middleware globally (via app.use()) so new routes are protected by default. Then add an ownership middleware that compares req.user.id to the requested resource's owner. For routes that don't need a userId param at all, derive it from the token.

// SAFE: Global auth + ownership verification middleware. const express = require('express'); const app = express(); // ─── Global auth — all routes require authentication by default ─── app.use('/api', authMiddleware); // Applied to ALL /api routes. // ─── Pattern 1: Derive userId from token (best for BOLA prevention) ─── app.get('/api/me/orders', async (req, res) => { // req.user.id comes from the verified JWT — not the URL. const orders = await db.getOrders(req.user.id); res.json(orders); }); // ─── Pattern 2: Ownership middleware for resource routes ─── function ownershipCheck(paramName) { return async (req, res, next) => { const resourceUserId = req.params[paramName]; if (req.user.id !== resourceUserId && req.user.role !== 'admin') { return res.status(403).json({ error: 'Forbidden' }); } next(); }; } app.get('/api/users/:userId/settings', ownershipCheck('userId'), // ✅ Verifies caller owns the resource. async (req, res) => { const settings = await db.getUserSettings(req.params.userId); res.json(settings); } ); // ─── Pattern 3: Query scoped to authenticated user at DB level ─── app.get('/api/documents/:docId', async (req, res) => { // WHERE clause includes req.user.id — impossible to access others'. const doc = await db.query( 'SELECT * FROM documents WHERE id = $1 AND owner_id = $2', [req.params.docId, req.user.id] ); if (!doc) return res.status(404).json({ error: 'Not found' }); res.json(doc); });
