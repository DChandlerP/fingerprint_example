Why Is This an Issue?

CWE-943 covers NoSQL Injection — a vulnerability where user-controlled input is used to build NoSQL queries without proper sanitization or type enforcement. Unlike traditional SQL injection which targets query strings, NoSQL injection often targets the JSON/BSON structure of the query itself.

This is particularly common in Node.js applications using MongoDB, where attackers can inject special operators (like $gt, $ne, or $where) to bypass authentication or extract sensitive data.

  • Operator Injection — using objects like {"$ne": null} to make a query true for any record.
  • Authentication Bypass — bypassing login checks by making the password comparison always true.
  • JavaScript Injection — exploiting the $where operator to execute arbitrary JavaScript on the database server.
  • Data Exfiltration — leaking hidden fields or other users' data by altering query logic.

WhileNoSQL databases don't use SQL, they are not "injection-proof". The primary defense is strict input validation, using schema-based ORMs (like Mongoose), and disabling server-side JavaScript execution features.

Examples of the Issue

Dangerous: NoSQL Operator Injection (Node.js / Express / MongoDB)

If an application passes an untrusted object (e.g., from req.body) directly into a MongoDB query, an attacker can substitute a string with an object containing operators like $ne (not equal).

// DANGEROUS: req.body used directly in query. app.post('/login', async (req, res) => { const { username, password } = req.body; // If an attacker sends: {"username": "admin", "password": {"$ne": null}} // The query becomes: { username: "admin", password: { $ne: null } } // This matches the admin user regardless of their actual password! const user = await db.collection('users').findOne({ username: username, password: password }); if (user) { res.send("Logged in!"); } else { res.status(401).send("Failed"); } });

Dangerous: JavaScript Injection using $where operator

The $where operator allows running arbitrary JavaScript on the MongoDB server. If user input is concatenated into this string, it's equivalent to classic SQL injection but with the full power of JavaScript.

// DANGEROUS: String concatenation in $where clause. app.get('/users/search', async (req, res) => { const name = req.query.name; // Attacker sends: ?name='; return true; var dummy=' // The query becomes: { $where: "this.name == ''; return true; var dummy=''" } // This returns EVERY user in the database. const users = await db.collection('users').find({ $where: `this.name == '${name}'` }).toArray(); res.json(users); });

Dangerous: JSON Query Injection in Spring Data MongoDB

In Java, using @Query with placeholders value can still be vulnerable if the query is structurally manipulated by the input or if the input is not properly typed by the driver.

// DANGEROUS: Potential structural injection. public interface UserRepository extends MongoRepository { // If the input is not strictly a string, it might be interpreted as an object. @Query("{ 'username' : ?0, 'password' : ?1 }") User findByCustomQuery(Object username, Object password); // Attacker passes a Map for password: Map.of("$ne", null) }

How to Resolve

Fix (JS): Sanitize inputs and enforce types (mongo-sanitize)

Strip any keys starting with $ from user input. The mongo-sanitize middleware is a common tool for this. Alternatively, always ensure inputs are cast to strings before using them in a query.

// SAFE: Sanitizing req.body to remove operators. const sanitize = require('mongo-sanitize'); app.post('/login', async (req, res) => { // Strips out any keys starting with "$" const username = sanitize(req.body.username); const password = sanitize(req.body.password); // Or explicitly cast to string: // const password = String(req.body.password); const user = await db.collection('users').findOne({ username: username, password: password }); // ... });

Fix (JS): Use Mongoose for Schema Enforcement

Mongoose (an ODM) provides automatic type casting based on your schema. If your schema says a field is a String, Mongoose will cast an injected object like {$ne: null} to the literal string "[object Object]", effectively neutralizing the attack.

// SAFE: Mongoose casts inputs to schema-defined types. const UserSchema = new mongoose.Schema({ username: String, password: String }); const User = mongoose.model('User', UserSchema); app.post('/login', async (req, res) => { // Even if req.body.password is {"$ne": null}, // Mongoose treats it as a string literal. const user = await User.findOne({ username: req.body.username, password: req.body.password }); // ... });

Fix (Config): Harden Database and avoid dangerous operators

Avoid the $where and $mapReduce operators entirely if possible. Most queries can be rewritten using standard MongoDB query operators. Additionally, disable server-side JavaScript execution in the MongoDB configuration.

// SAFE: Rewriting $where to standard query operators. // DANGEROUS: { $where: "this.name == '" + name + "'" } // SAFE: const users = await db.collection('users').find({ name: name // Simple, parameterized matching }).toArray(); // --- DATABASE CONFIGURATION --- // In mongod.conf, disable server-side JS execution: // security: // javascriptEnabled: false
