Why Is This an Issue?

CWE-297 flags code that connects to a remote host over TLS/SSL but does not verify that the server's certificate matches the hostname being requested. This is often caused by custom TrustManager, HostnameVerifier, or rejectUnauthorized: false configurations that silently accept any certificate — including self-signed, expired, or certificates issued to a completely different domain.

Without hostname verification, an attacker in a man-in-the-middle (MITM) position can present their own valid certificate (for their domain) and intercept all traffic — credentials, API keys, PII, session tokens — even though the connection appears to be "encrypted." The TLS handshake succeeds, the padlock icon shows, but the client is talking to the attacker, not the intended server.

  • Java — custom X509TrustManager with empty checkServerTrusted(), or HostnameVerifier that returns true for all hostnames
  • Node.js — setting rejectUnauthorized: false on HTTPS agents or NODE_TLS_REJECT_UNAUTHORIZED=0 in the environment
  • Axios / Fetch — passing custom agents that disable certificate validation to HTTP clients

This weakness maps to OWASP A07:2021 — Identification and Authentication Failures. The fix is straightforward: use the platform's default certificate validation and never override hostname checking in production code. If you need to test against self-signed certificates, use environment-scoped trust stores — not code changes.

Examples of the Issue

Dangerous: Trust-all TrustManager and HostnameVerifier (Java)

This is the most common pattern flagged by SAST tools in Java. A custom X509TrustManager with an empty checkServerTrusted() method accepts any certificate — self-signed, expired, revoked, or issued to a different host. Combined with a permissive HostnameVerifier, TLS is effectively downgraded to plaintext. This is often introduced during development to bypass self-signed cert errors and accidentally left in production.

// DANGEROUS: Accepts ANY certificate — TLS is effectively disabled. import javax.net.ssl.*; import java.security.cert.X509Certificate; public class ApiClient { static { try { // Trust manager that accepts everything — including attacker certs. TrustManager[] trustAll = new TrustManager[] { new X509TrustManager() { public X509Certificate[] getAcceptedIssuers() { return null; } public void checkClientTrusted(X509Certificate[] c, String t) { } public void checkServerTrusted(X509Certificate[] c, String t) { } // ↑ EMPTY — no validation at all. MITM trivial. } }; SSLContext sc = SSLContext.getInstance("TLS"); sc.init(null, trustAll, new java.security.SecureRandom()); HttpsURLConnection.setDefaultSSLSocketFactory(sc.getSocketFactory()); // Hostname verifier that accepts any hostname. HttpsURLConnection.setDefaultHostnameVerifier( (hostname, session) -> true // Always true = no host checking. ); } catch (Exception e) { throw new RuntimeException(e); } } } // Also dangerous — OkHttp with trust-all: OkHttpClient client = new OkHttpClient.Builder() .sslSocketFactory(sc.getSocketFactory(), (X509TrustManager) trustAll[0]) .hostnameVerifier((h, s) -> true) // Hostname check disabled! .build();

Dangerous: rejectUnauthorized: false and NODE_TLS_REJECT_UNAUTHORIZED=0 (Node.js)

Setting rejectUnauthorized: false on an HTTPS agent tells Node.js to accept any TLS certificate, including certificates for the wrong hostname, self-signed certs, and expired certs. The environment variable NODE_TLS_REJECT_UNAUTHORIZED=0 does the same thing globally for the entire process — affecting every outbound HTTPS connection, not just the one you intended.

// DANGEROUS: Disables ALL certificate validation for this agent. const https = require('https'); const agent = new https.Agent({ rejectUnauthorized: false // Accepts any cert — MITM trivial. }); https.get('https://api.example.com/data', { agent }, (res) => { // Connection "succeeds" even if attacker swaps the cert. }); // ALSO DANGEROUS: Global disable — affects ALL HTTPS connections. process.env.NODE_TLS_REJECT_UNAUTHORIZED = '0'; // Every https.request, fetch(), axios call in the process is now vulnerable. // ALSO DANGEROUS: Axios with custom agent. const axios = require('axios'); const response = await axios.get('https://api.example.com/data', { httpsAgent: new https.Agent({ rejectUnauthorized: false }) }); // ALSO DANGEROUS: node-fetch with agent. const fetch = require('node-fetch'); const res = await fetch('https://api.example.com/data', { agent: new https.Agent({ rejectUnauthorized: false }) });

Dangerous: Spring RestTemplate / WebClient with disabled SSL (Java)

In Spring applications, developers often disable SSL verification on RestTemplate or WebClient to work around self-signed certificates in dev/staging environments. This configuration frequently leaks into production via shared beans or profiles that aren't properly scoped. Once deployed, every outbound API call from the service is vulnerable to MITM.

// DANGEROUS: RestTemplate that accepts any certificate. import org.apache.http.conn.ssl.NoopHostnameVerifier; import org.apache.http.conn.ssl.TrustAllStrategy; import org.apache.http.ssl.SSLContextBuilder; @Bean public RestTemplate restTemplate() throws Exception { SSLContext sslContext = SSLContextBuilder.create() .loadTrustMaterial(TrustAllStrategy.INSTANCE) // Trusts ALL certs .build(); CloseableHttpClient httpClient = HttpClients.custom() .setSSLContext(sslContext) .setSSLHostnameVerifier(NoopHostnameVerifier.INSTANCE) // No host check .build(); return new RestTemplate(new HttpComponentsClientHttpRequestFactory(httpClient)); } // ALSO DANGEROUS: WebClient with InsecureTrustManagerFactory. import io.netty.handler.ssl.SslContextBuilder; import io.netty.handler.ssl.util.InsecureTrustManagerFactory; @Bean public WebClient webClient() { SslContext sslContext = SslContextBuilder.forClient() .trustManager(InsecureTrustManagerFactory.INSTANCE) // Trust everything! .build(); HttpClient httpClient = HttpClient.create() .secure(t -> t.sslContext(sslContext)); return WebClient.builder() .clientConnector(new ReactorClientHttpConnector(httpClient)) .build(); }

How to Resolve

Fix (Java): Use default SSL validation or a scoped custom truststore

The simplest fix is to remove the custom TrustManager and HostnameVerifier entirely — Java's default SSL stack validates certificate chains, expiration, and hostname matching out of the box. If you need to trust an internal CA (e.g., corporate PKI), load it into a custom TrustManagerFactory scoped to that specific connection — never set globals that affect all connections.

// SAFE: Use Java's default SSL validation — no custom TrustManager needed. import javax.net.ssl.HttpsURLConnection; import java.net.URL; public class ApiClient { /** Default HTTPS — validates cert chain, expiry, AND hostname. */ public String fetchData(String endpoint) throws Exception { URL url = new URL(endpoint); HttpsURLConnection conn = (HttpsURLConnection) url.openConnection(); // No custom SSLSocketFactory, no custom HostnameVerifier. // Java's defaults handle everything correctly. conn.setRequestMethod("GET"); return new String(conn.getInputStream().readAllBytes()); } } // ─── If you need to trust an internal/private CA ─── import javax.net.ssl.*; import java.security.KeyStore; import java.io.FileInputStream; public class InternalApiClient { private final SSLSocketFactory sslFactory; public InternalApiClient(String truststorePath, char[] password) throws Exception { // Load ONLY your internal CA — don't trust everything. KeyStore trustStore = KeyStore.getInstance("PKCS12"); try (var is = new FileInputStream(truststorePath)) { trustStore.load(is, password); } TrustManagerFactory tmf = TrustManagerFactory.getInstance( TrustManagerFactory.getDefaultAlgorithm() ); tmf.init(trustStore); SSLContext ctx = SSLContext.getInstance("TLS"); ctx.init(null, tmf.getTrustManagers(), null); this.sslFactory = ctx.getSocketFactory(); // Hostname verification is STILL enforced by default. } }

Fix (JS): Remove rejectUnauthorized: false, use CA bundles for internal certs

Remove rejectUnauthorized: false and NODE_TLS_REJECT_UNAUTHORIZED=0 from all production code. Node.js validates certificates by default — no configuration needed for public CAs. For internal or self-signed certificates, pass the CA certificate via the ca option or the NODE_EXTRA_CA_CERTS environment variable, which adds your CA to the trust chain without disabling validation.

// SAFE: Default Node.js HTTPS — validates cert chain + hostname. const https = require('https'); // No custom agent needed for public CAs — defaults are correct. https.get('https://api.example.com/data', (res) => { // Certificate chain, expiry, and hostname are all verified. }); // ─── For internal/self-signed CAs: supply the CA cert ─── const fs = require('fs'); const agent = new https.Agent({ // Trust your internal CA WITHOUT disabling validation. ca: fs.readFileSync('/path/to/internal-ca.pem'), // rejectUnauthorized defaults to true — leave it that way. }); https.get('https://internal-api.corp.example.com/data', { agent }, (res) => { // Validates against your internal CA AND checks hostname. }); // Axios equivalent: const axios = require('axios'); const response = await axios.get('https://internal-api.corp.example.com/data', { httpsAgent: new https.Agent({ ca: fs.readFileSync('/path/to/internal-ca.pem') }) }); // Or set the env var — adds your CA to the default bundle: // NODE_EXTRA_CA_CERTS=/path/to/internal-ca.pem node app.js

Fix (Spring): Profile-scoped SSL config with proper truststore

If your Spring service connects to internal APIs with private CAs, configure a profile-scoped truststore via Spring Boot's built-in SSL bundles (Spring Boot 3.1+) or application properties. This keeps SSL verification intact while trusting your internal CA. Never use TrustAllStrategy or NoopHostnameVerifier.

// SAFE: Spring Boot 3.1+ SSL bundles — truststore via config, not code. // application.yml: // spring: // ssl: // bundle: // jks: // internal-api: // truststore: // location: classpath:internal-ca-truststore.p12 // password: ${TRUSTSTORE_PASSWORD} // type: PKCS12 // Then reference the bundle in your RestClient / WebClient: import org.springframework.boot.ssl.SslBundles; @Bean public RestClient restClient(RestClient.Builder builder, SslBundles sslBundles) { return builder .baseUrl("https://internal-api.corp.example.com") .apply(b -> b.setSslBundle(sslBundles.getBundle("internal-api"))) .build(); // Hostname verification is enforced. Only the specified CA is trusted. } // ─── For older Spring Boot (pre-3.1): manual truststore bean ─── @Bean @Profile("!local") // Only active in non-local environments public RestTemplate restTemplate() throws Exception { KeyStore trustStore = KeyStore.getInstance("PKCS12"); try (var is = new ClassPathResource("internal-ca-truststore.p12").getInputStream()) { trustStore.load(is, truststorePassword.toCharArray()); } SSLContext sslContext = SSLContextBuilder.create() .loadTrustMaterial(trustStore, null) // null = default trust strategy .build(); CloseableHttpClient httpClient = HttpClients.custom() .setSSLContext(sslContext) // DEFAULT hostname verifier — do NOT use NoopHostnameVerifier! .build(); return new RestTemplate(new HttpComponentsClientHttpRequestFactory(httpClient)); }
