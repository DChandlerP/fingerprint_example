Why Is This an Issue?

CWE-1321 covers Prototype Pollution — a vulnerability unique to JavaScript (Node.js and Browser) where an attacker can modify the properties of the base Object prototype. Because almost every object in JavaScript inherits from this prototype, these changes "pollute" all objects in the application.

This occurs when user-controlled data is recursively merged into an object (e.g., via _.merge or a custom deep-copy function) without checking for the magic properties __proto__ or constructor.

  • Denial of Service (DoS) — Overwriting essential methods like toString, causing the application to crash.
  • Privilege Escalation — Injecting properties like isAdmin: true into the global prototype so all users are treated as admins.
  • Remote Code Execution (RCE) — Polluting configuration objects used by secondary libraries (e.g., template engines or child process spawners).

Prototype Pollution is a "source-to-sink" vulnerability. The source is usually a JSON merge, and the sink is the execution of a feature that depends on the polluted property.

Examples of the Issue

Dangerous: Unsafe Recursive Merge (JavaScript / Node.js)

Recursive merge functions that don't specifically block access to __proto__ allow an attacker to traverse to the global Object.prototype.

// DANGEROUS: Recursive merge without prototype checking. function merge(target, source) { for (let key in source) { if (typeof target[key] === 'object' && typeof source[key] === 'object') { merge(target[key], source[key]); // RECURSIVE CALL } else { target[key] = source[key]; } } } // Attacker Payload (JSON): // { "__proto__": { "isAdmin": true } } const userProfile = { name: "Bob" }; merge(userProfile, JSON.parse(attackerJson)); // Now EVERY object has isAdmin: true! const guest = {}; console.log(guest.isAdmin); // true

Dangerous: Prototype Pollution via URL Parameters

If the application uses a query parser (like qs with certain configurations) that allows object nesting, an attacker can pollute the prototype via the URL.

// DANGEROUS: Passing unvalidated query params to a merge. app.get('/settings', (req, res) => { const defaultSettings = { theme: 'dark' }; const userSettings = req.query.settings; // ?settings[__proto__][polluted]=true _.merge(defaultSettings, userSettings); // Old versions of Lodash/Underscore were vulnerable to this. });

How to Resolve

Fix (JS): Sanitize Keys and Block Magic Properties

Explicitly block the __proto__, constructor, and prototype keys in any merge or assignment logic involving user data.

// SAFE: Explicitly blocking prototype keys. function safeMerge(target, source) { for (let key in source) { // BLOCK DANGEROUS KEYS if (key === '__proto__' || key === 'constructor' || key === 'prototype') { continue; } if (typeof target[key] === 'object' && typeof source[key] === 'object') { safeMerge(target[key], source[key]); } else { target[key] = source[key]; } } }

Fix (JS): Objects without Prototypes

When creating objects that will hold user-controlled key-value pairs (like query parameters or map data), use Object.create(null). This creates an object that does not inherit from Object.prototype, making it impossible to pollute.

// SAFE: Create a map without a prototype. const safeMap = Object.create(null); // Even if someone pollutes Object.prototype, it won't affect safeMap. console.log(safeMap.__proto__); // undefined console.log(safeMap.toString); // undefined

Fix (Architecture): Input Validation via Schema

Use a library like ajv (JSON Schema) to validate the structure of incoming JSON objects. A strict schema that only allows expected fields will naturally reject __proto__ keys.

// SAFE: Validate JSON against a strict schema. const Ajv = require("ajv"); const ajv = new Ajv(); const schema = { type: "object", properties: { theme: { type: "string" } }, additionalProperties: false // ← This is critical }; const validate = ajv.compile(schema); if (validate(req.body)) { // Process data... } else { // Reject data... }
