Why Is This an Issue?

CWE-200 flags code that exposes sensitive information to actors who are not explicitly authorized to access it. This includes leaking stack traces, database connection strings, internal IP addresses, user PII, or application internals through error messages, HTTP headers, logs, or API responses.

Information exposure is dangerous because it gives attackers a detailed map of your system. A stack trace reveals framework versions, file paths, and database schemas. Verbose error messages disclose which fields exist in a query. Even seemingly harmless details like server software versions help attackers select the right exploit. This information turns a blind attack into a targeted one.

This weakness maps to OWASP A01:2021 — Broken Access Control. The fix is straightforward: return generic error messages to clients, log detailed diagnostics server-side only, and strip sensitive headers and metadata from production responses.

Examples of the Issue

Dangerous: Returning full stack traces to the client

This servlet catches exceptions and writes the entire stack trace to the HTTP response. The stack trace reveals the framework version, internal class names, file paths, line numbers, and the database driver — giving an attacker a detailed blueprint of the application's internals.

// DANGEROUS: Full stack trace sent to the client. @WebServlet("/user") public class UserServlet extends HttpServlet { protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException { try { String userId = req.getParameter("id"); User user = userDao.findById(userId); resp.getWriter().write(user.toJson()); } catch (Exception e) { // EXPOSED: Stack trace reveals internal paths, DB driver, and framework version. // e.g. "com.mysql.cj.jdbc.exceptions.CommunicationsException: // at com.myapp.dao.UserDao.findById(UserDao.java:42)" resp.setStatus(500); e.printStackTrace(resp.getWriter()); } } }

Dangerous: Express.js error handler returning full error object to client

This global error handler sends the complete error object — including the stack property and any attached metadata — directly in the JSON response. In development this is convenient, but in production it exposes file paths, dependency versions, and database query details.

// DANGEROUS: Full error details returned to the client. const express = require('express'); const app = express(); app.get('/api/users/:id', async (req, res, next) => { try { const user = await db.query('SELECT * FROM users WHERE id = $1', [req.params.id]); res.json(user.rows[0]); } catch (err) { next(err); } }); // Global error handler leaks everything. app.use((err, req, res, next) => { res.status(500).json({ message: err.message, // e.g. "relation 'users' does not exist" stack: err.stack, // Full file paths and line numbers query: err.query, // The raw SQL query code: err.code // e.g. "42P01" (PostgreSQL error code) }); });

Dangerous: Serializing entire entity objects — exposing internal fields

This Spring controller returns the full JPA entity directly. The entity includes fields like passwordHash, ssn, and internalNotes that are persisted in the database but should never be sent to the client. Jackson serializes every field by default.

// DANGEROUS: Returning the full JPA entity exposes sensitive fields. @Entity public class User { @Id private Long id; private String username; private String email; private String passwordHash; // EXPOSED to the client! private String ssn; // EXPOSED to the client! private String internalNotes; // EXPOSED to the client! private LocalDateTime createdAt; // ... getters and setters } @RestController public class UserController { @GetMapping("/api/users/{id}") public User getUser(@PathVariable Long id) { // Jackson serializes ALL fields — including passwordHash, ssn, internalNotes. return userRepository.findById(id) .orElseThrow(() -> new ResponseStatusException(HttpStatus.NOT_FOUND)); } }

How to Resolve

Fix (Java): Return generic messages to the client, log details server-side

Never send exception details to the client. Return a generic error message with a correlation ID that maps to your server-side logs. This gives operations teams full diagnostic context while revealing nothing to the attacker.

// SAFE: Generic error to client, full details in server logs only. @WebServlet("/user") public class UserServlet extends HttpServlet { private static final Logger log = LoggerFactory.getLogger(UserServlet.class); protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException { try { String userId = req.getParameter("id"); User user = userDao.findById(userId); resp.getWriter().write(user.toJson()); } catch (Exception e) { // Generate a correlation ID for traceability. String errorId = UUID.randomUUID().toString(); // Log the full exception server-side with the correlation ID. log.error("Error processing request [errorId={}]", errorId, e); // Return a generic message — no internal details. resp.setStatus(500); resp.setContentType("application/json"); resp.getWriter().write(String.format( "{\"error\": \"An internal error occurred.\", \"errorId\": \"%s\"}", errorId)); } } }

Fix (JS): Environment-aware error handler — suppress details in production

Use NODE_ENV to control error verbosity. In production, return only a generic message and correlation ID. In development, include the stack trace for debugging convenience. Always log the full error server-side.

// SAFE: Production errors hide internals; dev errors include stack traces. const { randomUUID } = require('crypto'); app.use((err, req, res, next) => { const errorId = randomUUID(); // Always log the full error server-side. console.error(`[${errorId}]`, err); const isProd = process.env.NODE_ENV === 'production'; res.status(err.status || 500).json({ error: 'An internal error occurred.', errorId, // Only include details in development. ...(isProd ? {} : { message: err.message, stack: err.stack }) }); });

Fix (Java): Use DTOs to explicitly control which fields are exposed

Never return JPA entities directly. Create a Data Transfer Object (DTO) that contains only the fields the client needs. This is an explicit allowlist — even if new sensitive columns are added to the database later, they won't appear in API responses unless deliberately added to the DTO.

// SAFE: DTO exposes only approved fields — passwordHash, ssn, etc. are excluded. public record UserDto(Long id, String username, String email, LocalDateTime createdAt) { /** Factory method to map entity → DTO. Sensitive fields are never included. */ public static UserDto from(User entity) { return new UserDto( entity.getId(), entity.getUsername(), entity.getEmail(), entity.getCreatedAt() // passwordHash, ssn, internalNotes are deliberately omitted. ); } } @RestController public class UserController { @GetMapping("/api/users/{id}") public UserDto getUser(@PathVariable Long id) { User user = userRepository.findById(id) .orElseThrow(() -> new ResponseStatusException(HttpStatus.NOT_FOUND)); // Only DTO fields are serialized — sensitive data never leaves the server. return UserDto.from(user); } // Also configure Spring's global exception handler for safety: @ExceptionHandler(Exception.class) public ResponseEntity<Map<String, String>> handleException(Exception e) { String errorId = UUID.randomUUID().toString(); log.error("Unhandled exception [errorId={}]", errorId, e); return ResponseEntity.status(500).body(Map.of( "error", "An internal error occurred.", "errorId", errorId )); } }
