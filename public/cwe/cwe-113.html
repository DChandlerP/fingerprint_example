Why Is This an Issue?

CWE-113 covers two related SAST findings:

  • Improper Neutralization — user input is placed into HTTP headers without stripping \r\n (CRLF) characters
  • Disabled Neutralization — the framework's built-in CRLF protection is explicitly turned off

HTTP headers are delimited by CRLF sequences (\r\n). If an attacker can inject these characters into a header value, they can split the HTTP response — inserting arbitrary headers or even a completely new response body. This enables:

  • HTTP Response Splitting — injecting a second response that the browser interprets as a separate page
  • Header Injection — adding headers like Set-Cookie to hijack sessions
  • Cache Poisoning — causing proxies and CDNs to cache malicious responses
  • XSS via response body injection — injecting HTML/JS after a double CRLF

This weakness maps to OWASP A03:2021 — Injection. Modern frameworks (Spring, Express, Servlet 4.0+) reject CRLF in headers by default — but older versions or misconfigured applications remain vulnerable. The fix is to validate and sanitize all user input before placing it in headers, and never disable built-in CRLF protections.

Examples of the Issue

Dangerous: User input placed directly in HTTP response header (Java)

This servlet reads a user-supplied lang parameter and sets it as a custom response header. If the attacker submits en\r\nSet-Cookie: session=hijacked, the CRLF characters terminate the current header and inject a malicious Set-Cookie header — hijacking the victim's session.

// DANGEROUS: User input injected into response header without sanitization. @WebServlet("/preferences") public class PreferencesServlet extends HttpServlet { protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException { String language = req.getParameter("lang"); // Attacker-controlled // If lang = "en\r\nSet-Cookie: session=hijacked\r\n\r\n<script>alert(1)</script>" // the response becomes: // X-User-Language: en // Set-Cookie: session=hijacked // // <script>alert(1)</script> resp.setHeader("X-User-Language", language); // CRLF injection! resp.getWriter().write("Preferences updated."); } }

Dangerous: User input in Location header via redirect (Node.js)

This Express route uses res.setHeader() to set the Location header from a user-supplied redirect parameter. Older versions of Node.js (before v14.18.0) did not block CRLF in setHeader(), allowing full response splitting. Even in newer versions, using raw HTTP writes bypasses these protections.

// DANGEROUS: User-controlled value in Location header. const express = require('express'); const app = express(); app.get('/redirect', (req, res) => { const target = req.query.url; // Attacker-controlled // Vulnerable on Node < 14.18.0 or if using raw response writes. // If url = "/home\r\nSet-Cookie: admin=true\r\n\r\nMalicious body" // the attacker injects headers AND a response body. res.setHeader('Location', target); res.status(302).end(); }); // Even worse — raw writes bypass ALL framework protections: app.get('/raw-redirect', (req, res) => { const target = req.query.url; // This completely bypasses Node's CRLF protections. res.socket.write( `HTTP/1.1 302 Found\r\nLocation: ${target}\r\n\r\n` ); res.socket.end(); });

Dangerous: Explicitly disabling the framework's CRLF header protection (Java)

Spring Security's HeaderWriterFilter includes CRLF protection by default. Some developers disable the header security filter entirely (or use a custom filter that skips validation) to work around formatting issues — re-opening the application to CRLF injection. This is the "disabled neutralization" variant that SAST tools flag.

// DANGEROUS: Disabling Spring Security's header protection. @Configuration @EnableWebSecurity public class SecurityConfig extends WebSecurityConfigurerAdapter { @Override protected void configure(HttpSecurity http) throws Exception { http // THIS DISABLES ALL SECURITY HEADERS including CRLF protection! .headers().disable() .authorizeRequests() .anyRequest().authenticated(); } } // Also dangerous — setting headers from user input in a Spring controller: @RestController public class ApiController { @GetMapping("/api/data") public ResponseEntity<String> getData(@RequestParam String format) { return ResponseEntity.ok() // User input directly in header — CRLF injection risk. .header("X-Response-Format", format) .body("{ \"data\": [] }"); } }

How to Resolve

Fix (Java): Strip CRLF characters and validate against an allowlist

Always strip or reject \r and \n from any user input before placing it in a header. Better yet, validate against an allowlist of expected values so that arbitrary strings are never reflected into headers at all.

// SAFE: Validate against allowlist + strip CRLF as defense in depth. @WebServlet("/preferences") public class PreferencesServlet extends HttpServlet { // Allowlist of valid language codes. private static final Set<String> VALID_LANGUAGES = Set.of("en", "es", "fr", "de", "ja", "zh"); protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException { String language = req.getParameter("lang"); // Step 1: Validate against allowlist (best defense). if (language == null || !VALID_LANGUAGES.contains(language)) { language = "en"; // Safe default. } // Step 2: Defense in depth — strip CRLF even from allowlisted values. language = sanitizeHeaderValue(language); resp.setHeader("X-User-Language", language); resp.getWriter().write("Preferences updated."); } /** Remove CR, LF, and null bytes from header values. */ private static String sanitizeHeaderValue(String value) { return value.replaceAll("[\\r\\n\\0]", ""); } }

Fix (JS): Validate redirect URLs and use framework APIs (never raw writes)

Use Express's res.redirect() (which includes CRLF protection in modern versions) instead of raw header writes. Validate redirect targets against an allowlist of domains, and never write directly to the socket to set headers.

// SAFE: Validate redirect target + use framework API. const { URL } = require('url'); const ALLOWED_HOSTS = new Set(['example.com', 'app.example.com']); app.get('/redirect', (req, res) => { const target = req.query.url; // Step 1: Parse and validate the URL. let parsedUrl; try { parsedUrl = new URL(target, `${req.protocol}://${req.get('host')}`); } catch { return res.status(400).send('Invalid URL.'); } // Step 2: Check against domain allowlist. if (!ALLOWED_HOSTS.has(parsedUrl.hostname)) { return res.status(400).send('Redirect target not allowed.'); } // Step 3: Use the framework's redirect — it handles CRLF protection. // Express (v4.17+) rejects CRLF in res.redirect() and res.setHeader(). res.redirect(302, parsedUrl.toString()); }); // NEVER do this — raw socket writes bypass all protections: // res.socket.write(`HTTP/1.1 302 Found\r\nLocation: ${target}\r\n\r\n`);

Fix (Java): Keep framework header protections enabled + centralized sanitization

Never disable Spring Security's header protection. If you need to customize specific headers, configure them individually rather than disabling the entire filter. Add a centralized sanitization utility for any dynamic header values.

// SAFE: Header protections ON + selective customization. @Configuration @EnableWebSecurity public class SecurityConfig extends WebSecurityConfigurerAdapter { @Override protected void configure(HttpSecurity http) throws Exception { http // DO NOT call .headers().disable()! // Customize individual headers instead: .headers() .contentTypeOptions().and() // X-Content-Type-Options .frameOptions().deny().and() // X-Frame-Options .xssProtection().and() // X-XSS-Protection .httpStrictTransportSecurity() .maxAgeInSeconds(31536000) .includeSubDomains(true) .and().and() .authorizeRequests() .anyRequest().authenticated(); } } // Centralized header value sanitizer for dynamic values: @Component public class HeaderSanitizer { private static final Pattern CRLF_PATTERN = Pattern.compile("[\\r\\n\\0]"); /** Sanitize a value before setting it as a response header. */ public String sanitize(String headerValue) { if (headerValue == null) return ""; String sanitized = CRLF_PATTERN.matcher(headerValue).replaceAll(""); // Truncate to prevent header size attacks. return sanitized.length() > 256 ? sanitized.substring(0, 256) : sanitized; } } // Usage in controller: @RestController public class ApiController { @Autowired private HeaderSanitizer headerSanitizer; @GetMapping("/api/data") public ResponseEntity<String> getData(@RequestParam String format) { // Map to allowlisted values when possible: String safeFormat = Set.of("json", "xml", "csv").contains(format) ? format : "json"; return ResponseEntity.ok() .header("X-Response-Format", headerSanitizer.sanitize(safeFormat)) .body("{ \"data\": [] }"); } }
