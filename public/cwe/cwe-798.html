Why Is This an Issue?

This finding covers three closely related CWEs that SAST tools frequently flag:

  • CWE-798 — Use of Hard-coded Credentials (usernames, passwords, API keys)
  • CWE-259 — Use of Hard-coded Password (a specific subset of 798)
  • CWE-547 — Use of Hard-coded, Security-relevant Constants (encryption keys, salts, tokens)

When credentials or secrets are embedded directly in source code, they become visible to anyone with access to the codebase — other developers, contractors, open-source consumers, or attackers who gain access to version control. Secrets in compiled binaries can be extracted with simple tools like strings. Once exposed, these credentials cannot be rotated without a code change and redeployment, creating a window of vulnerability.

CWE-798 is ranked in the SANS/CWE Top 25 and maps to OWASP A07:2021 — Identification and Authentication Failures. CWE-547 maps to OWASP A05:2021 — Security Misconfiguration. The fix is the same for all three: externalize secrets using environment variables, secrets managers, or vault services — never embed them in code.

Examples of the Issue

Dangerous: Hardcoded database password in Java source (CWE-798/259)

The database password is a string literal in the source code. It will be committed to version control history (even if later removed), compiled into the JAR, and visible to anyone who decompiles the class. Rotating this password requires a code change, rebuild, and redeployment.

// DANGEROUS: Database credentials hardcoded in source. public class DatabaseConfig { private static final String DB_URL = "jdbc:postgresql://prod-db:5432/myapp"; private static final String DB_USER = "admin"; private static final String DB_PASS = "SuperSecret123!"; // Hardcoded password! public static Connection getConnection() throws SQLException { return DriverManager.getConnection(DB_URL, DB_USER, DB_PASS); } }

Dangerous: Hardcoded API keys and JWT secret in Node.js (CWE-798/547)

This Express middleware uses a hardcoded string as the JWT signing secret and embeds a third-party API key directly in the source. If this repo is public (or becomes public accidentally), both secrets are immediately compromised. Even in private repos, every developer and CI/CD system has access.

// DANGEROUS: Secrets embedded directly in source code. const jwt = require('jsonwebtoken'); const axios = require('axios'); const JWT_SECRET = 'my-super-secret-jwt-key-2024'; // Hardcoded! const STRIPE_API_KEY = 'sk_live_EXAMPLE_FAKE_KEY_REPLACE_ME'; // Hardcoded! function verifyToken(req, res, next) { const token = req.headers.authorization?.split(' ')[1]; try { req.user = jwt.verify(token, JWT_SECRET); next(); } catch { res.status(401).json({ error: 'Invalid token' }); } } async function chargeCustomer(customerId, amount) { return axios.post('https://api.stripe.com/v1/charges', { customer: customerId, amount }, { headers: { Authorization: `Bearer ${STRIPE_API_KEY}` } }); }

Dangerous: Hardcoded AES encryption key and salt (CWE-547)

The AES key and salt are byte arrays defined directly in source. An attacker who obtains the JAR or source can extract these constants and decrypt any data encrypted by this class. Because the key is static, every deployment uses the same key — compromising one environment compromises all of them.

// DANGEROUS: Encryption key and salt hardcoded in source. public class EncryptionUtil { // Hardcoded AES key — extractable from compiled .class file. private static final byte[] AES_KEY = "MyS3cretK3y!2024".getBytes(StandardCharsets.UTF_8); private static final byte[] SALT = {0x01, 0x02, 0x03, 0x04, 0x05, 0x06, 0x07, 0x08}; public static byte[] encrypt(byte[] plaintext) throws Exception { SecretKeySpec keySpec = new SecretKeySpec(AES_KEY, "AES"); Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding"); cipher.init(Cipher.ENCRYPT_MODE, keySpec); return cipher.doFinal(plaintext); } }

How to Resolve

Fix (Java): Externalize credentials with Spring Boot properties + environment variables

Use Spring Boot's externalized configuration to read credentials from environment variables or a secrets file at runtime. The source code never contains the actual values. Environment variables can be set by the deployment platform (Kubernetes Secrets, AWS Parameter Store, etc.) without touching the codebase.

// SAFE: Credentials injected at runtime via environment variables. // application.yml — references env vars, never contains actual secrets: // spring: // datasource: // url: ${DB_URL} // username: ${DB_USERNAME} // password: ${DB_PASSWORD} @Configuration public class DatabaseConfig { @Value("${spring.datasource.url}") private String dbUrl; @Value("${spring.datasource.username}") private String dbUser; @Value("${spring.datasource.password}") private String dbPassword; @Bean public DataSource dataSource() { HikariConfig config = new HikariConfig(); config.setJdbcUrl(dbUrl); config.setUsername(dbUser); config.setPassword(dbPassword); return new HikariDataSource(config); } } // Set environment variables at deploy time (never in source): // export DB_URL="jdbc:postgresql://prod-db:5432/myapp" // export DB_USERNAME="admin" // export DB_PASSWORD="rotated-secret-from-vault"

Fix (JS): dotenv for local dev + cloud secrets manager for production

Use dotenv for local development (with .env in .gitignore) and a cloud secrets manager (AWS Secrets Manager, GCP Secret Manager, Azure Key Vault) for production. The code references variable names — never the values themselves.

// SAFE: Secrets loaded from environment, never in source code. require('dotenv').config(); // Loads .env file (in .gitignore!) for local dev. const jwt = require('jsonwebtoken'); // Read from environment — fails fast if missing. const JWT_SECRET = process.env.JWT_SECRET; const STRIPE_API_KEY = process.env.STRIPE_API_KEY; if (!JWT_SECRET || !STRIPE_API_KEY) { throw new Error('Missing required environment variables: JWT_SECRET, STRIPE_API_KEY'); } function verifyToken(req, res, next) { const token = req.headers.authorization?.split(' ')[1]; try { req.user = jwt.verify(token, JWT_SECRET); next(); } catch { res.status(401).json({ error: 'Invalid token' }); } } // .env file (NEVER committed to git): // JWT_SECRET=generated-random-256-bit-key // STRIPE_API_KEY=sk_live_... // // .gitignore must include: // .env // .env.local // .env.production

Fix (Java): Load encryption keys from a Key Management Service (KMS)

For encryption keys and salts, use a dedicated Key Management Service — AWS KMS, GCP Cloud KMS, Azure Key Vault, or HashiCorp Vault. The key material never exists in your source code or application memory in plaintext. The KMS performs encryption/decryption operations server-side, or provides keys that are rotated automatically.

// SAFE: Encryption key loaded from AWS KMS at startup — never in source. import software.amazon.awssdk.services.kms.KmsClient; import software.amazon.awssdk.services.kms.model.GenerateDataKeyRequest; import software.amazon.awssdk.services.kms.model.GenerateDataKeyResponse; public class EncryptionUtil { private static final String KMS_KEY_ARN = System.getenv("KMS_KEY_ARN"); private final KmsClient kmsClient; public EncryptionUtil() { this.kmsClient = KmsClient.create(); // Uses IAM role credentials. } /** Generate a unique data key for each encryption operation. */ public GenerateDataKeyResponse generateDataKey() { return kmsClient.generateDataKey(GenerateDataKeyRequest.builder() .keyId(KMS_KEY_ARN) .keySpec("AES_256") .build()); } public byte[] encrypt(byte[] plaintext) throws Exception { GenerateDataKeyResponse dataKey = generateDataKey(); // Use the plaintext key for encryption. SecretKeySpec keySpec = new SecretKeySpec( dataKey.plaintext().asByteArray(), "AES"); Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding"); cipher.init(Cipher.ENCRYPT_MODE, keySpec); byte[] ciphertext = cipher.doFinal(plaintext); // Store the ENCRYPTED data key alongside the ciphertext. // The plaintext key is never persisted — only KMS can decrypt it. return combineKeyAndCiphertext(dataKey.ciphertextBlob().asByteArray(), ciphertext); } }
