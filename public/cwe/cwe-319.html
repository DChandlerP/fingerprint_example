Why Is This an Issue?

CWE-319 flags code that transmits sensitive information — such as credentials, session tokens, personal data, or API keys — over an unencrypted channel. When data travels in cleartext (plain HTTP, unencrypted sockets, or FTP), anyone on the network path can intercept it: Wi-Fi eavesdroppers, compromised routers, ISPs, or attackers performing a man-in-the-middle (MITM) attack.

This is not a hypothetical risk. Tools like Wireshark and mitmproxy can capture and inspect unencrypted traffic in seconds. Public Wi-Fi networks, shared hosting environments, and misconfigured load balancers are common attack surfaces. Even internal traffic between services should be encrypted — a compromised internal host can sniff all unencrypted traffic on the same network segment.

This weakness maps to OWASP A02:2021 — Cryptographic Failures. The fix is to enforce TLS/HTTPS everywhere, redirect all HTTP traffic to HTTPS, and use Strict-Transport-Security (HSTS) headers to prevent downgrade attacks.

Examples of the Issue

Dangerous: Sending credentials over plain HTTP

This Java code posts login credentials to an http:// endpoint. The username and password travel across the network in plaintext — visible to any packet sniffer. Even if the server-side application is secure, the transport layer is not.

// DANGEROUS: Credentials sent over unencrypted HTTP. URL url = new URL("http://api.example.com/login"); // NOT https! HttpURLConnection conn = (HttpURLConnection) url.openConnection(); conn.setRequestMethod("POST"); conn.setDoOutput(true); conn.setRequestProperty("Content-Type", "application/json"); String body = String.format("{\"username\":\"%s\",\"password\":\"%s\"}", username, password); try (OutputStream os = conn.getOutputStream()) { os.write(body.getBytes(StandardCharsets.UTF_8)); } // The credentials are visible to any network observer between client and server. int status = conn.getResponseCode();

Dangerous: Express.js serving login form over HTTP without redirect

This Express server handles authentication over plain HTTP. The login form, session cookies, and all subsequent requests travel unencrypted. Even if the session cookie has httpOnly set, an attacker on the network can steal it via packet capture — httpOnly only prevents JavaScript access, not network interception.

// DANGEROUS: Serving authentication over plain HTTP — no TLS. const express = require('express'); const app = express(); app.use(express.urlencoded({ extended: true })); app.post('/login', (req, res) => { const { username, password } = req.body; // Cleartext on the wire! if (authService.verify(username, password)) { req.session.userId = username; res.redirect('/dashboard'); } else { res.status(401).send('Invalid credentials.'); } }); // Listening on HTTP — no TLS certificate, no encryption. app.listen(3000, () => console.log('Running on http://localhost:3000'));

Dangerous: Session cookie without the Secure flag

This servlet sets a session token cookie without the Secure flag. Even if the application is served over HTTPS, the browser will also send this cookie over plain HTTP requests (e.g., if the user visits http:// by mistake or an attacker forces a downgrade). The session token is then exposed in cleartext on the wire.

// DANGEROUS: Session cookie sent over both HTTP and HTTPS (no Secure flag). @WebServlet("/login") public class LoginServlet extends HttpServlet { protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException { String user = req.getParameter("username"); String pass = req.getParameter("password"); if (authService.authenticate(user, pass)) { String token = generateSessionToken(); Cookie sessionCookie = new Cookie("SESSION_ID", token); sessionCookie.setHttpOnly(true); sessionCookie.setPath("/"); // Missing: sessionCookie.setSecure(true); // The cookie will be sent over HTTP too — exposed to MITM. resp.addCookie(sessionCookie); resp.sendRedirect("/dashboard"); } } }

How to Resolve

Fix (Java): Enforce HTTPS with Spring Security + HSTS header

Configure Spring Security to require HTTPS for all requests and add the Strict-Transport-Security (HSTS) header. HSTS tells browsers to always use HTTPS for the domain, even if the user types http:// — preventing protocol downgrade attacks.

// SAFE: All HTTP requests are redirected to HTTPS; HSTS prevents downgrades. @Configuration @EnableWebSecurity public class SecurityConfig extends WebSecurityConfigurerAdapter { @Override protected void configure(HttpSecurity http) throws Exception { http // Redirect all HTTP requests to HTTPS. .requiresChannel() .anyRequest().requiresSecure() .and() // HSTS: browser will refuse HTTP for 1 year after first visit. .headers() .httpStrictTransportSecurity() .includeSubDomains(true) .maxAgeInSeconds(31536000) // 1 year .and().and() .authorizeRequests() .anyRequest().authenticated(); } } // application.properties — configure the TLS certificate: // server.port=8443 // server.ssl.key-store=classpath:keystore.p12 // server.ssl.key-store-password=${KEYSTORE_PASSWORD} // server.ssl.key-store-type=PKCS12

Fix (JS): HTTPS server + HTTP-to-HTTPS redirect + Helmet HSTS

Serve the application over HTTPS using Node's https module, run a minimal HTTP server that redirects to HTTPS, and use the helmet middleware to set HSTS and other security headers automatically.

// SAFE: HTTPS server with HTTP redirect and HSTS via Helmet. const https = require('https'); const http = require('http'); const fs = require('fs'); const express = require('express'); const helmet = require('helmet'); const app = express(); // Helmet sets Strict-Transport-Security and other security headers. app.use(helmet({ hsts: { maxAge: 31536000, // 1 year includeSubDomains: true, preload: true } })); // Configure secure session cookies. app.use(session({ secret: process.env.SESSION_SECRET, cookie: { secure: true, // Cookie only sent over HTTPS httpOnly: true, // Not accessible via JavaScript sameSite: 'lax' } })); // HTTPS server with TLS certificate. const httpsServer = https.createServer({ key: fs.readFileSync('/etc/ssl/private/server.key'), cert: fs.readFileSync('/etc/ssl/certs/server.crt') }, app); httpsServer.listen(443); // HTTP server that redirects everything to HTTPS. http.createServer((req, res) => { res.writeHead(301, { Location: `https://${req.headers.host}${req.url}` }); res.end(); }).listen(80);

Fix (Java): Always set the Secure flag on sensitive cookies

The Secure flag instructs the browser to only send the cookie over HTTPS. Combined with HttpOnly and SameSite, this provides defense in depth against both network interception and cross-site attacks. Always set all three flags on session and authentication cookies.

// SAFE: Cookie is only sent over HTTPS — never exposed in cleartext. @WebServlet("/login") public class LoginServlet extends HttpServlet { protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException { String user = req.getParameter("username"); String pass = req.getParameter("password"); if (authService.authenticate(user, pass)) { String token = generateSessionToken(); Cookie sessionCookie = new Cookie("SESSION_ID", token); sessionCookie.setSecure(true); // HTTPS only — never sent over HTTP sessionCookie.setHttpOnly(true); // Not accessible via JavaScript sessionCookie.setPath("/"); sessionCookie.setMaxAge(3600); // 1 hour // Set SameSite via Set-Cookie header (no direct API before Servlet 6.0). resp.setHeader("Set-Cookie", String.format("SESSION_ID=%s; Path=/; Secure; HttpOnly; SameSite=Lax; Max-Age=3600", token)); resp.sendRedirect("/dashboard"); } } }
