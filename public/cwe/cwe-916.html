Why Is This an Issue?

CWE-916 flags code that hashes passwords using algorithms that are too fast or lack sufficient computational effort to resist brute-force attacks. Unlike CWE-327 (which covers broken algorithms in general), CWE-916 specifically targets password hashing that uses a general-purpose hash — even a strong one like SHA-256 or SHA-512 — without an adaptive work factor. Common offenders include:

  • SHA-256 / SHA-512 — cryptographically strong, but designed for speed; billions of hashes/sec on a GPU makes brute-force trivial
  • Single-iteration PBKDF2 — the iteration count is the whole point; using 1 iteration is equivalent to a plain hash
  • MD5 / SHA-1 with a salt — adding a salt prevents rainbow tables but does not slow down brute-force attacks
  • Custom "stretch" loops — hand-rolled iteration loops are error-prone and lack the memory-hardness of modern algorithms

The core issue is speed. A password hash must be intentionally slow — taking hundreds of milliseconds per hash — so that offline brute-force attacks become computationally infeasible. General-purpose hashes are designed for the opposite: maximum throughput. Even with a salt, an attacker with a stolen database can test billions of candidate passwords per second using GPUs or ASICs.

This weakness maps to OWASP A02:2021 — Cryptographic Failures. The fix is to use a purpose-built password hashing algorithm with an adaptive work factor: bcrypt, Argon2id, or scrypt. If PBKDF2 is required (e.g., FIPS compliance), use at least 600,000 iterations with HMAC-SHA-256 as recommended by OWASP.

Examples of the Issue

Dangerous: SHA-256 used for password hashing (Java)

SHA-256 is a strong general-purpose hash, but it is far too fast for password storage. A single GPU can compute over 5 billion SHA-256 hashes per second. Adding a random salt prevents precomputed rainbow tables but does nothing to slow down brute-force attacks — an attacker can still crack most passwords in minutes.

// DANGEROUS: SHA-256 is too fast for password hashing. import java.security.MessageDigest; import java.security.SecureRandom; import java.util.HexFormat; public class UserService { /** Hash a password with a random salt using SHA-256. */ public String hashPassword(String password) throws Exception { byte[] salt = new byte[16]; new SecureRandom().nextBytes(salt); // SHA-256: strong for integrity, but ~5 billion hashes/sec on GPU. MessageDigest md = MessageDigest.getInstance("SHA-256"); md.update(salt); byte[] hash = md.digest(password.getBytes(StandardCharsets.UTF_8)); // Even with a salt, brute-force is trivial due to speed. return HexFormat.of().formatHex(salt) + ":" + HexFormat.of().formatHex(hash); } public boolean verifyPassword(String password, String stored) throws Exception { String[] parts = stored.split(":"); byte[] salt = HexFormat.of().parseHex(parts[0]); MessageDigest md = MessageDigest.getInstance("SHA-256"); md.update(salt); byte[] hash = md.digest(password.getBytes(StandardCharsets.UTF_8)); return parts[1].equals(HexFormat.of().formatHex(hash)); } }

Dangerous: PBKDF2 with insufficient iterations (Node.js)

PBKDF2 is designed to be slow via iteration count, but using too few iterations (e.g., 1,000 or even 10,000) defeats the purpose entirely. At 1,000 iterations, a GPU can still test millions of passwords per second. OWASP recommends a minimum of 600,000 iterations with HMAC-SHA-256, or 1,300,000 with HMAC-SHA-1. Many legacy systems were deployed with dangerously low iteration counts that have never been updated.

// DANGEROUS: Iteration count is far too low. const crypto = require('crypto'); function hashPassword(password) { const salt = crypto.randomBytes(16); // 1,000 iterations was acceptable in 2004 — it is not in 2025. // At this count, PBKDF2 offers almost no brute-force resistance. const hash = crypto.pbkdf2Sync( password, salt, 1000, // WAY too low — minimum should be 600,000 64, // key length 'sha256' ); return salt.toString('hex') + ':' + hash.toString('hex'); } // Also dangerous at 10,000 iterations: const hash = crypto.pbkdf2Sync(password, salt, 10000, 64, 'sha256'); // Even 100,000 is below OWASP's current recommendation: const hash2 = crypto.pbkdf2Sync(password, salt, 100000, 64, 'sha1');

Dangerous: Spring NoOpPasswordEncoder and MessageDigestPasswordEncoder (Java)

Spring Security's NoOpPasswordEncoder stores passwords in plain text, and MessageDigestPasswordEncoder(deprecated) uses a single-iteration hash like SHA-256 or MD5. Both are flagged by SAST tools because they provide zero or near-zero computational resistance to brute-force attacks. These encoders exist only for legacy compatibility and testing — they must never be used in production.

// DANGEROUS: No hashing at all — passwords stored in plain text. import org.springframework.security.crypto.password.NoOpPasswordEncoder; @Configuration public class SecurityConfig { @Bean public PasswordEncoder passwordEncoder() { // NoOpPasswordEncoder: stores passwords AS-IS. Deprecated for a reason. return NoOpPasswordEncoder.getInstance(); } } // ALSO DANGEROUS: Single-iteration SHA-256 via deprecated encoder. import org.springframework.security.crypto.password.MessageDigestPasswordEncoder; @Bean public PasswordEncoder passwordEncoder() { // One round of SHA-256 — billions of hashes/sec on GPU. return new MessageDigestPasswordEncoder("SHA-256"); } // ALSO DANGEROUS: Manual hash in a custom UserDetailsService. @Override public UserDetails loadUserByUsername(String username) { // Fetching a SHA-256 hashed password from the database. String storedHash = userRepository.findHashByUsername(username); return new User(username, storedHash, authorities); // Password was hashed with MessageDigest — no work factor, no protection. }

How to Resolve

Fix (Java): Use bcrypt or Argon2id with an adaptive cost factor

Replace SHA-256/SHA-512 with bcrypt (cost 12+) or Argon2id. Both are purpose-built for passwords: they are intentionally slow, include a built-in salt, and have a tunable work factor that can be increased as hardware improves. Argon2id is the winner of the Password Hashing Competition and is resistant to both GPU and side-channel attacks due to its memory-hardness.

// SAFE: bcrypt is purpose-built for password hashing. // Add dependency: org.mindrot:jbcrypt:0.4 import org.mindrot.jbcrypt.BCrypt; public class UserService { // Cost factor 12 = ~250ms per hash — makes brute-force impractical. // Increase to 13 or 14 as hardware improves. private static final int BCRYPT_COST = 12; /** Hash a password for storage. */ public String hashPassword(String password) { return BCrypt.hashpw(password, BCrypt.gensalt(BCRYPT_COST)); } /** Verify a password against its stored bcrypt hash. */ public boolean verifyPassword(String password, String storedHash) { return BCrypt.checkpw(password, storedHash); } } // ─── Alternative: Argon2id (recommended for new projects) ─── // Add dependency: de.mkammerer:argon2-jvm:2.11 import de.mkammerer.argon2.Argon2; import de.mkammerer.argon2.Argon2Factory; public class UserService { private static final Argon2 argon2 = Argon2Factory.create(Argon2Factory.Argon2Types.ARGON2id); /** Hash using Argon2id: memory-hard + CPU-hard. */ public String hashPassword(String password) { // Parameters: iterations=3, memory=64MB, parallelism=1 return argon2.hash(3, 65536, 1, password.toCharArray()); } public boolean verifyPassword(String password, String storedHash) { return argon2.verify(storedHash, password.toCharArray()); } }

Fix (JS): bcrypt for new projects, PBKDF2 with ≥600K iterations for FIPS

Use bcrypt (salt rounds 12+) for new applications. If FIPS 140-2 compliance requires PBKDF2, use at least 600,000 iterations with HMAC-SHA-256 per OWASP 2023 guidance. The iteration count should be re-evaluated annually and increased as hardware improves — treat it as a configuration value, not a hard-coded constant.

// SAFE: bcrypt with sufficient cost factor. const bcrypt = require('bcrypt'); const SALT_ROUNDS = 12; // ~250ms per hash — tuneable. async function hashPassword(password) { return bcrypt.hash(password, SALT_ROUNDS); } async function verifyPassword(password, storedHash) { return bcrypt.compare(password, storedHash); } // ─── FIPS-compliant alternative: PBKDF2 with 600K+ iterations ─── const crypto = require('crypto'); // OWASP 2023: minimum 600,000 iterations with HMAC-SHA-256. const PBKDF2_ITERATIONS = 600_000; const KEY_LENGTH = 64; const DIGEST = 'sha256'; function hashPasswordPBKDF2(password) { const salt = crypto.randomBytes(16); const hash = crypto.pbkdf2Sync( password, salt, PBKDF2_ITERATIONS, KEY_LENGTH, DIGEST ); // Store iteration count alongside hash for future upgrades. return `${PBKDF2_ITERATIONS}:${salt.toString('hex')}:${hash.toString('hex')}`; } function verifyPasswordPBKDF2(password, stored) { const [iterations, saltHex, hashHex] = stored.split(':'); const salt = Buffer.from(saltHex, 'hex'); const hash = crypto.pbkdf2Sync( password, salt, parseInt(iterations), KEY_LENGTH, DIGEST ); // Constant-time comparison to prevent timing attacks. return crypto.timingSafeEqual(hash, Buffer.from(hashHex, 'hex')); }

Fix (Spring): BCryptPasswordEncoder or DelegatingPasswordEncoder

Replace NoOpPasswordEncoder and MessageDigestPasswordEncoder with Spring's BCryptPasswordEncoder (cost 12+). For applications that need to migrate legacy hashes, use DelegatingPasswordEncoder — it can verify old formats while re-hashing to bcrypt on the next successful login.

// SAFE: Spring BCryptPasswordEncoder with cost 12. import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder; @Configuration public class SecurityConfig { @Bean public PasswordEncoder passwordEncoder() { // Strength 12 = cost factor 2^12 = ~250ms per hash. return new BCryptPasswordEncoder(12); } } // ─── Migration strategy: DelegatingPasswordEncoder ─── // Supports verifying legacy SHA-256 hashes while upgrading to bcrypt. import org.springframework.security.crypto.factory.PasswordEncoderFactories; @Bean public PasswordEncoder passwordEncoder() { // Default = bcrypt; also decodes {sha256}, {scrypt}, {argon2}. return PasswordEncoderFactories.createDelegatingPasswordEncoder(); } // Re-hash on successful login to transparently upgrade legacy passwords: public void onLoginSuccess(String username, String rawPassword) { User user = userRepository.findByUsername(username); String currentHash = user.getPasswordHash(); // If current hash is not bcrypt, re-hash with the new encoder. if (!currentHash.startsWith("$2a$") && !currentHash.startsWith("$2b$")) { String upgraded = passwordEncoder().encode(rawPassword); user.setPasswordHash(upgraded); userRepository.save(user); } }
